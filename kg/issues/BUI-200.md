## 목표

provider failure taxonomy가 실제 사용자 조치와 연결되도록 `last_error`, `last_failure`, status/report 출력을 정리한다.
실패 유형별로 사용자가 다음에 할 일을 확인할 수 있게 하고, raw provider 오류나 민감정보 후보는 사용자-facing 메시지에 노출하지 않는 것이 목표였다.

## 구현 내용

- `src/providers/failure.js`에 kind별 기본 `user_message`, `action`, `retryable`, `blocked` guidance를 추가하고 `interrupted` kind를 taxonomy에 포함했다.
- `auth`, `config`, `sandbox`, `timeout`, `model_response`, `interrupted`, `provider_unavailable`, `runner_normalize`, `runner_io`, `unknown` 경로가 표준 `failure.action`을 갖도록 정리했다.
- `scripts/status.js`는 `state.last_failure` 또는 `progress.last_failure`의 `action`을 `next_action`으로 출력하게 했다.
- `scripts/report.js`, `src/progress-writer.js`, `src/providers/standard-writer.js`는 실패 result Markdown에 다음 조치를 함께 쓰고, `last_error`는 사용자-facing safe message를 우선 기록하게 했다.
- `progress.json.last_failure`와 `state.json.last_failure`에는 `kind`, `code`, `retryable`, `blocked`, `action`만 남기고 `debug_detail`은 제외하는 경계를 유지했다.
- provider raw error, secret 후보, 홈 경로 후보는 `failure.debug_detail`에 sanitize 후 보관하고 status/report/result 같은 사용자-facing 출력에는 노출하지 않도록 회귀 테스트를 보강했다.

## 결정 사항

- `last_error`는 하위 호환 문자열로 유지한다. 구조화된 최신 실패 정보는 `last_failure`에 두고, 사용자 조치는 `last_failure.action`으로 노출한다.
- `failure.action`은 status/report/result Markdown에서 사용자가 바로 볼 수 있는 한글 다음 조치 문장으로 둔다.
- `debug_detail`은 내부 진단용이다. 사용자-facing 메시지와 조치 문장에는 raw provider 메시지, secret 후보, private environment value, 실제 홈 경로를 넣지 않는다.
- `failure.user_message`가 있는 표준 provider error 이벤트에서는 `progress.last_error`도 raw event message보다 `failure.user_message`를 우선한다.
- Claude stream-json terminal result가 error인데 provider failure 객체가 누락된 경우 writer가 표준 `model_response` failure를 합성해 raw result 노출을 막는다.

## 결정 이유

- 사용자는 `provider 실행 실패` 같은 단일 메시지만으로 인증, 설정, sandbox, timeout, 모델 응답 문제 중 무엇을 조치해야 하는지 알 수 없다.
- `retryable`과 `blocked`는 orchestration 판단 필드이고, 사람이 실제로 실행할 조치는 별도 `action` 문장이 필요하다.
- `last_error`를 객체로 바꾸면 기존 status/file contract 소비자가 깨질 수 있어 문자열 의미를 유지해야 했다.
- raw provider 메시지는 디버그에는 필요하지만 secret 후보와 로컬 경로를 포함할 수 있으므로 사용자-facing 경로와 분리해야 했다.
- writer가 provider failure 누락 경로를 보완하지 않으면 Claude error result가 그대로 `progress.last_error`나 `do-result.md`에 흘러갈 수 있다.

## 재발 방지 포인트

- provider failure kind를 추가하거나 classifier를 바꾸면 `user_message`, `action`, `retryable`, `blocked` 기본값을 함께 정한다.
- `last_error`는 safe user message만 담고, raw provider 문자열은 sanitize된 `debug_detail` 또는 로그 계층에만 둔다.
- `progress.json.last_failure`와 `state.json.last_failure`에는 `debug_detail`을 넣지 않는다.
- status/report/result Markdown 출력 변경 시 secret 후보와 실제 홈 경로가 사용자-facing 경로에 남지 않는 테스트를 같이 갱신한다.
- Claude terminal error result처럼 provider failure가 누락될 수 있는 경로는 writer fallback failure 합성 테스트를 유지한다.
