## 목표

provider 실행이 장시간 실행, 네트워크 지연, 사용자 중단, app-server turn 실패를 만날 때 무한 retry나 조용한 실패 없이 명확하게 끝나도록 한다.
provider별 timeout 기본값과 override 규칙을 문서화하고, `AbortSignal`/interrupt 전달, retry 가능 오류 분류, terminal event 이후 추가 이벤트 차단을 검증한다.

## 구현 내용

- `docs/contracts/provider-config.md`에 timeout/interrupt/retry 정책을 추가했다.
  Claude는 `MULTICA_AGENT_TIMEOUT` 또는 30분, Codex는 `timeout_ms` 또는 30분을 기본 timeout으로 사용한다.
- provider config에 `max_retries`, `retry_delay_ms`를 추가하고 숫자형 유효성 검증을 테스트로 고정했다.
- `src/pipeline-runner.js`가 phase별 provider config의 `signal`, `max_retries`, `retry_delay_ms`를 Codex adapter까지 전달하도록 연결했다.
- `src/providers/codex.js`에 retry wrapper를 추가했다.
  retry 가능한 중간 attempt의 이벤트는 buffer에 보관하고, 최종 attempt의 success/error만 `onEvent`로 flush한다.
- Codex adapter가 `AbortSignal` early-abort와 실행 중 abort listener를 모두 처리하도록 했다.
  interrupt는 `failure.kind=interrupted`, `retryable=false`, `blocked=false`로 terminal error를 남긴다.
- `src/providers/failure.js`에 `interrupted` failure kind와 Codex classifier를 추가했다.
- `providerMeta.retry`에 attempts, max_retries, retry log를 남겨 retry 횟수와 이유를 추적할 수 있게 했다.

## 결정 내용

- 자동 retry는 provider 내부의 bounded retry로 제한한다.
  `max_retries`가 0보다 클 때만 동작하고, retry 대상은 `failure.retryable=true`인 실패로 한정한다.
- timeout과 broker busy는 retry 가능 실패로 보고, auth/config/sandbox/interrupted는 즉시 실패로 둔다.
- 중간 retry attempt의 error event는 terminal event로 기록하지 않는다.
  state/progress/result 계약은 최종 attempt의 결과만 반영한다.
- 사용자 interrupt는 retryable 실패가 아니라 명시적 취소 결과다.
  따라서 `interrupted`는 blocked도 retryable도 아닌 terminal failure로 분류한다.

## 결정 이유

- 중간 attempt error를 그대로 writer에 흘리면 성공 retry 후에도 terminal error가 먼저 기록되어 state/progress/result 계약이 모순될 수 있다.
- retry 가능 여부와 자동 retry 실행 여부를 분리해야 한다.
  taxonomy의 `retryable=true`는 분류 신호이고, 실제 retry는 provider config의 cap과 attempt 수가 함께 결정한다.
- interrupt를 timeout이나 provider unavailable로 섞으면 사용자 취소와 인프라 실패를 구분할 수 없다.
- retry log를 providerMeta에 남기면 사용자 표시 파일 계약을 깨지 않고 운영자가 attempt history를 확인할 수 있다.

## 재발 방지 포인트

- retry wrapper를 수정할 때 중간 attempt의 buffered error가 `onEvent`로 flush되지 않는지 먼저 확인한다.
- terminal `error` 또는 `phase_end` 이후에는 같은 attempt에서 추가 이벤트를 기록하지 않는다.
- `interrupted`는 timeout 재시도 정책에 포함하지 않는다.
  사용자 취소는 retry 대상이 아니라 즉시 종료 신호다.
- provider config에 새 retry 필드를 추가할 때는 parser validation과 adapter 전달 테스트를 함께 갱신한다.
- retry 로그에는 attempt, kind/code, reason 같은 운영 판단 필드만 남기고 secret, token, private environment value는 남기지 않는다.
