## 목표

Codex provider adapter가 실제 app-server broker를 사용할 때 시작, endpoint 전달, 종료, stale process 정리, timeout cleanup이 안정적으로 동작하도록 정리한다.
사용자에게 보이는 실패도 인증 문제와 broker lifecycle 문제를 구분할 수 있어야 한다.

## 구현 내용

- `src/providers/codex.js`에 broker session lifecycle을 추가했다.
  `.built/runtime/codex-broker.json`에 endpoint, pid, pidFile, logFile, sessionDir, startedAt을 저장하고 `.built/runtime/codex-broker.lock`으로 중복 시작을 막는다.
- `BUILT_CODEX_BROKER_ENDPOINT`가 있으면 외부 broker endpoint를 우선 사용하고, 연결 불가 시 broker lifecycle 오류로 반환한다.
- 기존 session endpoint가 살아 있으면 재사용하고, stale session은 shutdown/unlink/rmdir cleanup 후 새 broker를 시작한다.
- broker app-server는 JSON-RPC line protocol socket으로 연결하고, 기존 direct spawn 경로는 `_disableBroker` fallback으로 유지했다.
- timeout 후 내부 broker session이면 cleanup을 수행하고, cleanup 실패는 result의 `cleanupError`로 남긴다.
- broker busy, broker start 실패, cleanup 실패 메시지를 분리해 lifecycle 오류가 인증/availability 오류와 섞이지 않게 했다.
- `test/providers-codex.test.js`에 fake broker lifecycle 테스트를 추가했다.
  endpoint 생성/파싱, session 시작과 state 저장, stale cleanup, cleanup 실패 반환, broker 경유 정상 실행, timeout 이후 session state 제거와 후속 실행 가능성을 검증한다.

## 결정 사항

- Codex provider는 기본적으로 broker lifecycle을 사용하되 `_disableBroker`로 기존 direct spawn 경로를 유지한다.
  broker 재사용을 도입하면서도 기존 app-server 단위 테스트와 fallback 검증 경로를 보존하기 위해서다.
- broker session state는 project cwd의 `.built/runtime` 아래에 둔다.
  workspace별 broker 재사용과 stale cleanup 판단을 같은 경계에서 처리할 수 있고, 산출물 계약 파일과 분리된다.
- startup lock은 exclusive create와 stale lock 감지로 처리한다.
  완전한 process supervisor가 없는 현재 구조에서 동시 시작을 줄이고, 죽은 process의 lock이 후속 실행을 영구 차단하지 않게 하기 위한 선택이다.
- timeout cleanup은 외부 broker endpoint에는 적용하지 않는다.
  외부 endpoint는 provider가 소유한 process가 아니므로 내부 session cleanup과 같은 책임으로 종료하면 안 된다.
- cleanup 실패는 조용히 무시하지 않고 `cleanupError`로 resolve한다.
  terminal event ordering 계약은 유지하면서도 사용자가 lifecycle cleanup 문제를 후속 조치할 수 있게 하기 위해서다.

## 재발 방지 포인트

- broker session state는 endpoint 생존 여부를 확인한 뒤 재사용해야 한다. state 파일 존재만으로 재사용하면 stale socket/PID가 후속 실행을 막을 수 있다.
- timeout cleanup은 provider가 소유한 내부 broker에만 적용한다. `BUILT_CODEX_BROKER_ENDPOINT`로 들어온 외부 broker는 종료 책임이 외부에 있다.
- cleanup 실패를 무시하면 다음 실행에서 같은 stale state가 반복된다. 실패 상세는 `cleanupError`나 명확한 broker 시작 오류로 남겨야 한다.
- lock 획득 실패는 단순 인증 실패가 아니다. broker busy/startup lifecycle 문제로 분리해 사용자가 재시도와 인증 조치를 구분할 수 있게 해야 한다.
- broker 경로 변경 시 direct spawn fallback과 provider terminal event ordering 계약을 함께 회귀 테스트해야 한다.
