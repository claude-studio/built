## 목표

provider 비교 모드 설계와 ADR-12를 실제 실행 가능한 MVP로 연결한다.
기본 `/built:run` 또는 `node scripts/run.js` 경로와 분리된 명시적 비교 명령에서만 Claude/Codex 등 candidate 결과를 같은 입력 snapshot과 acceptance criteria로 비교할 수 있게 한다.

## 구현 내용

- `src/providers/comparison-config.js`를 추가해 `run-request.json`의 top-level `comparison` 필드를 비교 명령 전용으로 파싱한다.
- MVP phase는 `do`만 허용하고, `comparison.enabled: true`가 아니면 비교 설정을 비활성으로 취급한다.
- `scripts/compare-providers.js`를 추가해 candidate별 branch, worktree, output directory를 분리하는 비교 전용 entrypoint를 만들었다.
- 비교 산출물은 `.built/runtime/runs/<feature>/comparisons/<comparison-id>/` 아래에 저장한다.
  candidate별 결과는 `providers/<candidate-id>/` 하위에 `run-request.json`, `state.json`, `progress.json`, `verification.json`, `diff.patch`, `git-status.txt`, `result/do-result.md`, `logs/do.jsonl` 형태로 남긴다.
- 공통 입력 evidence는 `input-snapshot.json`, `acceptance-criteria.md`, `verification-plan.json`으로 기록한다.
- 전체 결과는 `manifest.json`과 `report.md`로 요약하며, `report.md`에는 자동 winner를 선택하지 않았다는 문구를 남긴다.
- `BUILT_COMPARE_FAKE_PROVIDER=1` fake provider 모드를 추가해 실제 git worktree/provider 호출 없이 offline E2E 테스트를 실행할 수 있게 했다.
- `test/compare-providers.test.js`를 추가해 parser 단위 테스트와 fake E2E를 기본 테스트 suite에 포함했다.

## 결정 내용

- 비교 모드는 기본 runner에 통합하지 않고 `node scripts/compare-providers.js <feature> [--phase do] [--comparison <id>]`로만 실행한다.
- `providers.<phase>`는 계속 단일 provider 선택 계약으로 유지하고, 비교 대상은 top-level `comparison.candidates[]`에 둔다.
- canonical `.built/features/<feature>/`와 기본 `.built/runtime/runs/<feature>/state.json`은 비교 모드가 쓰지 않는다.
  비교 결과는 comparison output root 아래에만 저장한다.
- 자동 winner 선택, 자동 merge, canonical branch 적용은 MVP 범위에서 제외한다.
  비교 report와 diff는 사람이 판단할 evidence로 남긴다.
- real provider 비교는 후속 opt-in smoke로 남기고, 기본 회귀 테스트는 fake/offline 경로로 유지한다.

## 결정 이유

- 기본 실행과 비교 실행을 섞으면 phase별 provider routing 계약이 흐려지고 기존 `/built:run` 동작의 하위 호환성이 약해진다.
- candidate별 worktree/output을 분리해야 한 provider의 변경, diff, 로그가 다른 provider 결과나 canonical 산출물을 오염시키지 않는다.
- fake E2E는 인증, 네트워크, 비용, local provider availability에 의존하지 않으므로 기본 테스트 신호를 안정적으로 유지한다.
- 자동 winner 선택은 검증 명령 통과 여부만으로 acceptance criteria 해석 차이와 유지보수성을 판단하기 어렵기 때문에 아직 운영 정책으로 승인하지 않는다.

  real-mode 활성화 전 `git worktree add -b <branch> <worktreePath> <base_ref>` 단일 명령으로 수정해야 한다.
- 위 real-mode bug는 fake mode 완료 기준과 기본 E2E에는 영향을 주지 않지만, 실제 Claude/Codex 비교 smoke 전에는 blocker로 취급한다.

## 재발 방지 포인트

- comparison 관련 parser와 runner를 수정할 때 기본 `scripts/run.js` 경로에 comparison activation을 추가하지 않는다.
- 비교 산출물은 반드시 `.built/runtime/runs/<feature>/comparisons/<comparison-id>/providers/<candidate-id>/` 아래에 쓴다.
- canonical `.built/features/<feature>/` 보호 테스트를 유지한다.
  비교 실행이 canonical `do-result.md`, `progress.json`, 기본 run `state.json`을 만들거나 덮어쓰면 회귀로 본다.
- real provider 비교를 기본 테스트에 넣지 않는다.
  real smoke는 인증, 네트워크, 비용, provider availability에 의존하므로 명시적 opt-in으로만 실행한다.
- real-mode 활성화 전 candidate worktree 생성은 `git worktree add -b <branch> <worktreePath> <base_ref>` 방식으로 고쳐 project root HEAD 전환을 피한다.
- 비교 실행 후 cleanup은 설계 문서의 순서를 따른다.
  candidate worktree 제거, compare branch 삭제, comparison output root는 audit evidence로 보존한다.
