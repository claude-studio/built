## 목표

worktree-first run의 canonical 산출물이 execution worktree 내부 `.built/features/<feature>`에 생성된 뒤 cleanup 과정에서 archive 없이 유실될 수 있는 경로를 막는다.

## 구현 내용

- `scripts/cleanup.js`가 `--archive` 실행 시 worktree 제거 전에 registry `resultDir`, state `execution_worktree.result_dir`, root fallback `.built/features/<feature>` 후보 중 실제 존재하는 result dir를 archive 대상으로 선택하도록 수정했다.
- registry `resultDir`가 stale이거나 누락되어도 state `execution_worktree.result_dir` 산출물을 archive 최상위에 보존하도록 했다.
- worktree canonical result dir와 root fallback이 모두 존재하면 worktree result dir를 archive 최상위 산출물로 두고 root fallback은 `_root-fallback/` 아래에 별도 보존한다.
- `--archive` cleanup의 dirty 검사에서 canonical result dir 내부 산출물은 built-owned artifact로 허용하고, result dir 밖 uncommitted 변경은 계속 cleanup을 중단하도록 분리했다.
- `test/cleanup.test.js`에 worktree result dir 보존, stale registry pointer fallback, 기본 git 상태의 untracked result artifact 허용, result dir 밖 dirty 변경 차단 회귀 테스트를 추가했다.
- `docs/ops/artifact-retention-policy.md`에 worktree-first result dir 보존 정책, `--archive` 필요 조건, result artifact dirty 예외와 사용자 변경 보호 경계를 명시했다.

## 결정 내용

- cleanup archive의 canonical source는 단일 pointer가 아니라 registry `resultDir`, state `execution_worktree.result_dir`, root fallback 후보를 실제 존재 여부 기준으로 순회해 정한다.
- worktree result dir와 root fallback이 충돌하면 worktree result dir가 archive 최상위 우선이다.
- root fallback은 삭제하지 않고 `_root-fallback/`에 보존해 legacy 산출물과 최신 worktree 산출물을 구분한다.
- `--archive` cleanup은 canonical result dir 내부 untracked 산출물을 사용자 dirty 변경으로 보지 않는다.
- result dir 밖 변경은 archive 여부와 무관하게 cleanup을 중단한다.

## 결정 이유

- worktree-first run의 실제 산출물 위치는 root feature dir가 아니라 `state.execution_worktree.result_dir`일 수 있으므로 root fallback만 archive하면 evidence 유실이 발생한다.
- registry와 state pointer는 운영 중 불일치하거나 stale일 수 있다. cleanup은 status/cost와 같은 소비자처럼 실제 존재하는 canonical 후보를 보수적으로 찾아야 한다.
- result artifact는 built가 생성한 evidence이므로 archive 전에 dirty safety gate로 cleanup이 중단되면 보존 목표를 달성하지 못한다.
- 반대로 result dir 밖 uncommitted 변경은 사용자 작업일 수 있으므로 cleanup이 제거하면 안 된다.

## 재발 방지 포인트

- cleanup archive는 registry `resultDir` 하나만 믿지 말고 state `execution_worktree.result_dir`와 root fallback 후보를 함께 확인한다.
- worktree result dir 내부의 `report.md`, `do-result.md`, `check-result.md`, `logs/`, `progress*`는 built-owned artifact로 archive 대상이다.
- root fallback과 worktree result dir가 함께 있으면 `_root-fallback/` 정책을 유지해 어느 산출물이 canonical인지 모호하게 덮어쓰지 않는다.
- dirty safety gate를 수정할 때는 result artifact 보존과 사용자 변경 보호를 같은 테스트에서 분리해 검증한다.
- `--archive` 없는 cleanup은 여전히 result dir 삭제 대상이므로 감사나 handoff evidence가 필요하면 cleanup 전에 반드시 archive 옵션을 사용한다.
