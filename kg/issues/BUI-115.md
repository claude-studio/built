## 목표

phase별로 Claude/Codex provider를 선택할 수 있도록 요청 설정의 `providers` 필드를 파싱한다.
이 작업은 이후 `plan_synthesis`, `do`, `check` 확장에서 phase별 provider 선택을 가능하게 만드는 기반이다.

## 구현 내용

- `src/providers/config.js` 신규 추가: `parseProviderConfig`와 `getProviderForPhase` 구현.
- 단축형 설정 지원: `{ "providers": { "do": "codex" } }`.
- 상세형 설정 지원: `{ "providers": { "do": { "name": "codex", "model": "gpt-5.5", "sandbox": "workspace-write" } } }`.
- `providers` 필드가 없거나 특정 phase 설정이 없으면 기존 동작과 동일하게 `{ name: "claude" }`를 반환.
- provider 이름은 `claude`, `codex`만 허용하고, 알 수 없는 provider는 즉시 실패.
- `do`, `iter` phase에서 `claude` 외 provider가 `read-only` sandbox로 설정되면 명확한 오류를 반환.
- `test/providers-config.test.js` 신규 추가: 단축형, 상세형, 기본값, fallback, 잘못된 provider, sandbox 정책을 포함한 단위 테스트 32개 작성.

## 결정 사항

- `providers`가 없을 때 모든 phase 기본 provider는 `claude`로 유지한다.
  기존 요청 파일과 현재 Claude 기반 실행 동작을 깨지 않기 위해서다.
- `do`, `iter` phase는 파일 쓰기 가능성이 있으므로 `claude` 외 provider에 `workspace-write` sandbox를 요구한다.
  `read-only`로 조용히 성공하거나 중간에 애매하게 실패하는 것을 막기 위해 parser 단계에서 실패시킨다.
- `check`, `report`, `plan_synthesis`는 현재 정책상 `read-only` sandbox를 허용한다.
- parser는 순수 함수로 유지한다. spawn mock이나 실제 provider 실행 없이 설정 계약만 검증할 수 있어야 한다.

## 발생한 이슈

  - `providers` 필드가 없을 때 `parseProviderConfig`는 빈 맵을 반환하고, `getProviderForPhase`가 `{ name: "claude" }` 기본값을 반환한다.

## 재발 방지 포인트

- provider 설정은 조용히 무시하지 않고 허용 provider, 허용 필드, sandbox 정책을 parser 단계에서 검증한다.
- 기존 요청 파일 하위 호환은 `providers` 부재 테스트로 계속 고정한다.
- `do`, `iter`처럼 파일 쓰기 가능성이 있는 phase는 provider별 sandbox 기본값과 실패 메시지를 함께 검증한다.
- `src/pipeline-runner.js` 또는 `run.js`에서 parser를 실제 요청 흐름에 연결할 때도 기존 Claude fallback 테스트를 유지한다.
