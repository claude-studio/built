## 목표

provider config의 phase key 또는 ProviderSpec field 오타와 malformed `run-request.json`을 조용히 무시하지 않고, 실행 전에 config path와 원인을 드러내는 hard-fail 계약으로 고정한다.
`plan_synthesis` 같은 opt-in phase가 실제로 활성화되었는지도 stdout과 artifact에서 확인 가능하게 만든다.

## 구현 내용

- `src/providers/config.js`에서 `SUPPORTED_PHASES` 기반 phase allowlist와 ProviderSpec 필드 allowlist를 적용했다.
- `providers.plan_synthsis`, `providers.<phase>.typo_model`처럼 알 수 없는 phase 또는 field가 있으면 parser 단계에서 `providers.<phase>` 경로를 포함한 오류로 실패하게 했다.
- `scripts/run.js`에서 `run-request.json` JSON parse 오류와 provider config parse 오류를 run 진입 초기에 검증하고, path와 원인을 출력한 뒤 phase script 실행 전에 중단하게 했다.
- `plan_synthesis` 활성화 여부를 stdout의 `plan_synthesis: enabled|disabled`와 `state.json`의 `plan_synthesis_enabled` 필드로 기록했다.
- `docs/contracts/provider-config.md`에 지원 phase key, 허용 ProviderSpec field, unsupported key 실패 계약을 반영했다.
- `test/providers-config.test.js`와 `test/run.test.js`에 unknown phase, unknown field, malformed JSON/YAML-like request, unsupported phase, plan_synthesis 관측 회귀 테스트를 추가했다.

## 결정 내용

- `providers` map의 phase key는 `plan_synthesis`, `do`, `check`, `iter`, `report`만 허용한다.
- ProviderSpec 객체의 필드는 `name`, `model`, `timeout_ms`, `max_retries`, `retry_delay_ms`, `sandbox`, `effort`, `output_mode`만 허용한다.
- 설정 오류는 Claude fallback 또는 `plan_synthesis=false`로 복구하지 않는다. `/built:run` 진입 시점에 config path와 오류 원인을 보여주고 실패한다.
- `plan_synthesis_enabled`는 provider output이 아니라 runner state 관측 필드로 둔다.

## 결정 이유

- provider routing 오타가 조용히 무시되면 사용자는 의도한 provider나 opt-in phase가 실행되지 않은 상태를 성공으로 오해할 수 있다.
- `plan_synthesis`는 비용과 실패 모드가 있는 opt-in phase이므로 활성화 여부가 실행 로그와 durable artifact에 남아야 dogfooding 결과를 해석할 수 있다.
- parser 단계의 allowlist는 provider capability registry와 phase contract를 SSOT로 쓰게 해, 새 phase나 field 추가 시 명시적인 문서와 테스트 갱신을 요구한다.
- runner가 `run-request.json` parse error를 숨기면 malformed request가 legacy request처럼 취급되어 원인 분석이 늦어진다.
