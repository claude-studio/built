## 목표

do phase에서 Codex provider를 선택할 수 있는 최소 MVP를 추가한다.
BUI-117(Codex adapter), BUI-118(plan_synthesis phase) 패턴을 그대로 do phase에 연결하고,
fake provider 기반 E2E가 기본 테스트에서 통과하며 실제 Codex smoke는 opt-in으로 분리한다.

## 구현 내용

### 선행 조건 확인

- `pipeline-runner.js` — `providerSpec.name='codex'` 경로가 BUI-117에서 이미 구현되어 있었음.
  `runCodex` + `createStandardWriter` 조합으로 do phase 이벤트를 progress.json / do-result.md에 기록.
- `scripts/do.js` — BUI-118에서 `run-request.json`의 `providers.do` 설정을 읽어 `providerSpec`을 `runPipeline`에 전달하도록 이미 수정되어 있었음.
- `src/providers/config.js` — `do` phase provider config parsing 완비.
- E2E 시나리오 04 (`test/e2e/scenarios/04-fake-provider-file-contracts.js`) — fake Codex 이벤트 시퀀스로 do phase file contract 검증 완비.

### 추가한 내용

**`test/pipeline-runner.test.js` — Codex do path 단위 테스트 3개 추가**

- `mockSpawnSync` 헬퍼: `childProcess.spawnSync`를 임시 대체해 checkAvailability/checkLogin mock 지원.
- `makeFakeCodexAppServer` 헬퍼: stdin JSON-RPC 요청에 응답하는 가짜 app-server 프로세스 생성.
  `initialize → thread/start → turn/start → turn/completed` 알림 시퀀스를 시뮬레이션.
- 테스트 1: Codex 바이너리 없음 → `success:false` 반환.
- 테스트 2: `sandbox=workspace-write` 정상 실행 → `progress.json(status=completed)` + `do-result.md` 생성.
- 테스트 3: `sandbox=read-only` + `phase=do` → `success:false`, error 메시지에 `workspace-write` 언급.

**`scripts/smoke-codex-do.js` — opt-in 실제 Codex do smoke 스크립트**

- `BUILT_CODEX_DO_SMOKE=1` 환경 변수로만 실행. 미설정 시 skip 메시지 출력 후 exit 0.
- 임시 projectRoot 생성 → feature spec 작성 → `run-request.json(providers.do.name=codex, sandbox=workspace-write)` 작성.
- `scripts/do.js <feature>` 실행.
- 완료 후 `do-result.md` frontmatter 필수 필드(`feature_id, status, model, cost_usd, duration_ms, created_at`) 검증.
- `progress.json status=completed` 검증.
- 실패 시 원인 메시지 명시 출력 (바이너리 미설치, 인증 실패, 산출물 구조 불일치 구분).
- `BUILT_KEEP_SMOKE_DIR=1`로 임시 디렉토리 보존 가능.

## 결정 사항

**do phase Codex 경로는 추가 구현 없이 기존 pipeline-runner.js 경로를 그대로 사용**

- BUI-117에서 `pipeline-runner.js` Codex 경로가 완전히 구현되어 있었고,
  BUI-118에서 `do.js`가 이미 `providerSpec`을 읽도록 수정되어 있었다.
- 실제 do phase 파일 변경은 `codex` sandbox=`workspace-write`가 필수이므로,
  `do.js`가 넘기는 `providerSpec.sandbox`가 `workspace-write`인지 `providers/config.js`가 검증한다.
- plan_synthesis와 달리 do phase에는 별도 JSON schema 정규화 레이어가 없다.
  Codex가 실제 파일을 직접 작성(workspace-write sandbox)하고, runner는 이벤트 스트림만 기록한다.

**sandbox 정책 명시**

- `providers/config.js` — `do/iter` phase에서 `read-only` sandbox는 파싱 시 에러.
- `providers/codex.js` — `runCodex`에서 `phase=do/iter + sandbox=read-only` 조합은 즉시 error 이벤트 emit.
- 두 레이어 모두 검증하여 조용한 실패 방지.

## 발생한 이슈

없음. 선행 이슈(BUI-117, BUI-118)가 완료된 상태에서 do phase 경로가 이미 연결되어 있었다.
추가 작업은 누락된 단위 테스트와 opt-in smoke 스크립트 두 가지였다.

## 재발 방지 포인트

- **do phase + Codex = sandbox=workspace-write 필수**: `read-only`로 설정하면 Codex가 성공 응답을 보내도 실제 파일 변경이 없다. `providers/config.js` 파싱 단계와 `runCodex` 실행 단계 두 곳에서 검증한다.
- **pipeline-runner.js Codex 경로 테스트 패턴**: `childProcess.spawnSync`(availability/login)와 `childProcess.spawn`(app-server) 모두 mock 필요. providers-codex.test.js의 `_spawnFn`/`_spawnSyncFn` 주입 방식은 pipeline-runner 레벨에서 사용할 수 없어 모듈 객체 프로퍼티 패치 방식을 사용했다.
- **fake app-server 응답 순서**: initialize(id=1) → thread/start(id=2) → turn/start(id=3) → turn/started 알림 → item/completed 알림 → turn/completed 알림. id는 클라이언트 `nextId` 초기값 1에서 시작.
