## 목표

Claude와 Codex provider가 같은 입력에서 동일한 파일 계약을 만족하는지 오프라인 테스트로 검증한다.
provider별 raw event 차이는 표준 provider event로 normalize하고, `state.json`, `progress.json`, phase result markdown의 필수 구조가 provider와 무관하게 유지되는지 확인한다.

## 구현 내용

- `src/providers/event-normalizer.js` 신규 추가:
  - Claude stream-json raw event를 `phase_start`, `text_delta`, `tool_call`, `tool_result`, `usage`, `phase_end`, `error` 표준 이벤트로 변환한다.
  - Codex는 표준 이벤트를 직접 emit한다는 가정 아래 passthrough와 timestamp 보완을 수행한다.
  - `checkOrderingRules`, `checkToolPairing`으로 terminal 이후 이벤트 금지와 tool call/result pairing을 검증한다.
- `src/providers/standard-writer.js` 신규 추가:
  - 표준 이벤트를 받아 `.built/features/<feature>/progress.json`과 `do-result.md`를 기록한다.
  - `progress-writer.js`와 동일한 필수 필드와 파일 위치 계약을 따른다.
- `test/providers-normalizer.test.js` 신규 추가:
  - provider event normalize, ordering, tool pairing 단위 테스트를 추가했다.
- `test/e2e/scenarios/04-fake-provider-file-contracts.js` 신규 추가:
  - fake Claude와 fake Codex provider를 같은 입력으로 실행해 provider별 산출물 필수 구조 동일성을 검증한다.

## 결정 사항

- 표준 이벤트는 계약 문서 기준으로 `usage`를 비용/토큰 optional metric 이벤트로 사용한다.
  이슈 설명에는 `cost`가 표준 이벤트 중 하나로 적혀 있었지만, `docs/contracts/provider-events.md`는 `usage` 안에 `cost_usd`, `input_tokens`, `output_tokens`를 담도록 정의한다.
- Codex normalizer는 실제 Codex app-server 연결 전까지 표준 이벤트 passthrough로 둔다.
  실제 notification 형식이 확정되면 BUI-117에서 매핑을 보완해야 한다.
- provider는 파일을 직접 쓰지 않고 표준 이벤트만 전달한다.
  파일 위치와 필수 필드 동일성은 `standard-writer`와 E2E fixture가 검증한다.
- 한 phase를 Claude와 Codex가 동시에 처리하는 기본 동작은 추가하지 않았다.
  비교 실행은 fake provider 기반 테스트 fixture로만 다뤘다.

## 발생한 이슈

- Builder 완료 보고 시 `npm test` 전체 통과가 확인되었다.
- Review Round 1에서 Pass.
- Reviewer 확인 사항:
  - 신규 파일 4개 추가만 있었고 기존 파일 수정은 없었다.
  - fake provider E2E는 Node.js 내장 모듈만 사용하므로 CI에서 오프라인 실행 가능하다.
  - `progress.json` 필수 키는 `feature`, `phase`, `session_id`, `turn`, `tool_calls`, `last_text`, `cost_usd`, `input_tokens`, `output_tokens`, `started_at`, `updated_at`, `status`로 고정되었다.
  - `error` 이후 별도 `phase_end`를 허용하지 않는 terminal 규칙과 tool call/result pairing 검증이 포함되었다.
- 비차단 참고:
  - 이슈 본문에는 표준 이벤트를 `cost`로 표현했지만 구현과 계약 문서는 `usage`를 사용한다.
  - `onUsage`는 비용을 누적하고 `onPhaseEnd`는 비용을 덮어쓴다. 실제 비용 검증은 비범위이며 BUI-117 실제 Codex 연결 시 재검토한다.

## 재발 방지 포인트

- provider event 타입 이름은 이슈 텍스트가 아니라 `docs/contracts/provider-events.md`를 기준으로 삼는다.
- provider별 raw event를 writer에 직접 연결하지 말고 반드시 표준 이벤트 normalize 계층을 거친다.
- `phase_end` 또는 `error` 이후 이벤트를 허용하면 progress/result 상태가 뒤집힐 수 있으므로 terminal rule 테스트를 유지한다.
- fake provider E2E는 실제 provider 연결 전에도 파일 위치와 필수 키 계약을 고정하는 회귀 방지 장치다.
- 실제 Codex 연결 시 passthrough 가정, usage/cost 누적 정책, notification 타입 매핑을 다시 검증한다.
