## 목표

사용자가 `/built:run-codex` 계열 실행을 Ctrl-C, SIGTERM, `/built:abort <feature>`, timeout으로 중단한 뒤에도 Codex app-server active turn이 계속 파일을 생성하거나 수정하는 위험을 제거한다.
`state.json.status=aborted`인 상태와 실제 provider 작업이 살아 있는 상태가 모순되지 않도록 active turn metadata와 cleanup 결과를 기록한다.

## 구현 내용

- phase script 공통 중단 helper `src/phase-abort.js`를 추가하고 `scripts/do.js`, `scripts/check.js`, `scripts/iter.js`, `scripts/report.js`, `scripts/plan-synthesis.js`, `scripts/run.js`가 `AbortSignal`을 `runPipeline()`에 전달하도록 연결했다.
- Codex active turn metadata helper `src/codex-active-turn.js`를 추가해 `.built/runtime/runs/<feature>/state.json`과 `.built/features/<feature>/progress.json`에 `active_provider.threadId`, `turnId`, `phase`, `cwd`, `status`, `interrupt`를 기록한다.
- `src/providers/codex.js`가 `turn/start` 응답 또는 `turn/started` notification 이후 `provider_metadata.active_provider`를 emit하고, signal/timeout 경로에서 `turn/interrupt`를 시도한 결과를 `codex_interrupt`로 terminal `error`에 싣도록 보강했다.
- `scripts/abort.js`는 state, registry, lock cleanup을 먼저 완료한 뒤 저장된 Codex active turn metadata로 `interruptCodexTurn()`을 호출한다.
  interrupt 호출은 bounded timeout을 갖고, 실패 또는 timeout이면 `active_provider.status=interrupt_failed`와 `codex_interrupt.interrupted=false`를 기록하며 수동 종료 안내를 출력한다.
- `src/pipeline-runner.js`와 `src/providers/standard-writer.js`가 `provider_metadata`와 `codex_interrupt`를 최종 progress/state에 보존하도록 했다.
- 외부 `/built:abort`가 이미 `aborted`를 기록한 경우 parent runner가 후속 phase 성공/실패 정리로 상태를 덮지 않도록 방어했다.
- `docs/contracts/provider-events.md`에 `provider_metadata`, `active_provider`, `codex_interrupt` 이벤트 계약을 추가했다.

## 결정 내용

- Codex active turn의 `threadId`/`turnId`는 외부 abort가 읽을 수 있는 runtime state와 feature progress에 함께 기록한다.
  progress만 있으면 runtime cleanup이 놓칠 수 있고, state만 있으면 사용자-facing 최종 상태와 drift가 날 수 있기 때문이다.
- `/built:abort`는 state/registry/lock cleanup을 Codex `turn/interrupt` 응답보다 먼저 끝낸다.
  app-server가 응답하지 않아도 사용자의 중단 의도와 lock 해제는 보존되어야 한다.
- interrupt 실패를 성공처럼 숨기지 않는다.
  `codex_interrupt`와 `active_provider.status=interrupt_failed`를 남겨 "작업이 아직 계속될 수 있음"을 사용자와 후속 역할이 확인할 수 있게 한다.
- `provider_metadata`는 provider의 파일 소유권이 아니라 runner/control plane의 관측 및 중단 제어 이벤트다.
  결과 파일 normalization 책임은 계속 `standard-writer`와 runner에 둔다.

## 재발 방지 포인트

- detached broker를 유지하더라도 active turn cleanup은 별도 계약으로 반드시 수행해야 한다.
  broker process lifecycle과 user abort lifecycle을 같은 것으로 간주하면 중단 후 파일 변경이 재발한다.
- `/built:abort`는 app-server 응답성에 의존해 state/registry/lock cleanup을 늦추면 안 된다.
  interrupt는 bounded timeout으로 처리하고 실패 metadata를 남긴다.
- 외부 abort가 기록한 `aborted` 상태는 parent runner의 후속 phase 정리보다 우선한다.
- `provider_metadata`를 provider 파일 쓰기 통로로 사용하지 않는다.
  control-plane metadata는 runner/writer가 normalize하고 사용자-facing 파일 계약은 provider 무관하게 유지한다.
- interrupt 실패 detail에는 운영 판단에 필요한 요약만 남기고 secret, token, private environment value, raw execution dump는 남기지 않는다.
