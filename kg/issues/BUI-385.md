## 목표

`scripts/report.js`의 KG draft 생성 위치를 helper plugin root가 아니라 target project root 기준으로 정렬한다.
설치형 plugin package/cache에서 실행해도 target project의 `kg/issues/<FEATURE>.md`에 draft가 생성되고, plugin root에는 target artifact를 쓰지 않아야 한다.

## 구현 내용

- `scripts/report.js`가 feature spec과 KG draft target root를 `BUILT_PROJECT_ROOT` 또는 현재 target cwd 기준으로 계산하도록 변경했다.
- `generateKgDraft()`에 `targetRoot` 입력을 추가하고, legacy `pluginRoot`는 fallback으로만 쓰도록 했다.
- report artifact frontmatter에 `kg_draft`, `kg_draft_target_root`를 기록한다.
- Run `root-context.json`의 `artifact_paths.kg_draft`에 target project 기준 KG draft 예상 경로를 남긴다.
- `docs/contracts/file-contracts.md`에 Report KG draft 경로가 target project root 기준이라는 계약을 추가했다.
- 설치형 plugin fixture 테스트를 추가해 target project와 plugin root가 다른 환경에서 target KG draft 생성과 plugin root 미작성을 검증했다.

## 결정 내용

- Report KG draft의 target root는 `BUILT_PROJECT_ROOT`가 있으면 그 값을 우선하고, 없으면 target cwd 기준으로 계산한다.
- `pluginRoot`는 legacy fallback과 helper code provenance 의미로만 남긴다. 설치형 plugin package/cache root는 target project artifact 저장 기준이 아니다.
- `root-context.json`의 `artifact_paths.kg_draft`와 report frontmatter의 `kg_draft`, `kg_draft_target_root`는 같은 target project 기준 경로를 가리켜야 한다.
- target project에 `kg/issues`가 없으면 기존 `generateKgDraft()` 정책처럼 skip하되, skip 판단도 plugin root가 아니라 target root 기준으로 한다.

## 결정 이유

- 설치형 plugin에서는 helper `__dirname`이 target project가 아니라 plugin package/cache를 가리킬 수 있다. 이 값을 draft root로 쓰면 report는 성공해도 target project에 KG draft가 보이지 않는다.
- root-context 계약은 `project_root`와 `plugin_root`를 분리한다. Report KG draft도 이 계약을 따라야 dogfooding repo와 설치형 target project의 동작이 같아진다.
- report artifact와 root-context에 target path를 남기면 실패 분석자가 전체 실행 로그를 보지 않고도 draft 생성 위치와 root drift 여부를 확인할 수 있다.

## 재발 방지 포인트

- Report, Run, Plan/Design 같은 target artifact writer는 helper `__dirname`이나 plugin package/cache root를 target project root로 사용하지 않는다.
- Report KG draft 경로를 바꿀 때는 `docs/contracts/file-contracts.md`, `root-context.json` `artifact_paths.kg_draft`, report frontmatter, fixture 테스트를 함께 갱신한다.
- 설치형 plugin fixture는 target project `kg/issues` 생성과 plugin root `kg/issues` 미작성을 동시에 확인해야 한다.
- target project에 `kg/issues`가 없을 때의 skip 정책은 유지하되, skip 여부를 판단하는 root가 target project인지 확인한다.
