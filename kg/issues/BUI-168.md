## 목표

provider smoke나 실제 Codex 실행 전에 사용자가 로컬 환경 문제를 한 번에 확인할 수 있는 diagnostics 진입점을 추가한다.
점검은 실제 모델 호출 없이 Codex CLI 설치, `codex app-server` 지원, 인증 상태, broker endpoint와 lock, run-request provider 설정, feature registry 경합 후보를 구분해 보여야 한다.

## 구현 내용

- `src/providers/doctor.js`를 추가해 provider 환경 사전 점검 로직을 모듈화했다.
- `scripts/provider-doctor.js`를 추가해 `node scripts/provider-doctor.js`, `npm run doctor`, `--json`, `--cwd`, `--feature` 진입점을 제공했다.
- 점검 결과는 `ok`/`warn`/`fail` 상태와 한글 `message`/`action`으로 표현하고, `--json`에서는 `overall`과 `checks[]` 구조로 출력한다.
- Codex 설치와 app-server 지원은 `src/providers/codex.js`의 `checkAvailability()`를 재사용하고, 인증은 `checkLogin()`을 재사용한다.
- run-request provider 설정은 `parseProviderConfig()`로 검증해 기존 provider config 계약과 같은 parser를 사용한다.
- broker session은 `_loadBrokerSession()` 결과의 PID 생존 여부와 unix socket 파일 존재 여부를 분리해 stale 후보를 `warn`으로 표시한다.
- broker lock은 파싱 실패, stale PID, 30초 초과 lock을 `warn`으로 표시하며 자동 삭제하지 않는다.
- feature registry에 running feature가 있으면 broker 경합 가능성을 `warn`으로 안내한다.
- `test/provider-doctor.test.js`를 추가해 실제 Codex CLI 없이 mock `spawnSync` 기반으로 29개 분기를 검증했다.
- `docs/ops/provider-setup-guide.md`에 provider doctor 사용법과 smoke 전 사전 점검 위치를 연결했다.

## 결정 내용

- doctor는 smoke를 대체하지 않고 사전 점검 전용 명령으로 둔다.
  doctor는 CLI, auth, broker state, config를 확인하고 smoke는 실제 provider 실행과 산출물 검증을 담당한다.
- stale broker session과 stale lock은 doctor가 삭제하지 않고 `warn`과 조치 메시지만 제공한다.
  실제 정리 책임은 기존 broker lifecycle 경로와 사용자의 명시 조치에 둔다.
- run-request provider 설정은 별도 ad hoc validator를 만들지 않고 `parseProviderConfig()`를 재사용한다.
- 출력은 raw stderr나 debug dump가 아니라 사용자 메시지와 권장 조치 중심으로 제한한다.
  JSON 출력도 `id`, `status`, `label`, `message`, `action`만 포함한다.

## 결정 이유

- 사전 점검 명령이 실제 모델 호출을 수행하면 smoke와 doctor의 책임 경계가 흐려지고 인증, 네트워크, 비용 의존성이 기본 diagnostics에 들어온다.
- broker stale 후보는 false positive 가능성이 있으므로 doctor 단계에서 자동 삭제하면 실행 중인 다른 작업을 훼손할 수 있다.
- 기존 provider config parser를 재사용해야 runner와 doctor가 같은 설정을 다르게 해석하는 회귀를 막을 수 있다.
- 사용자-facing diagnostics에 raw provider stderr, token 후보, private environment value를 싣지 않아야 ADR-9의 메시지 경계와 redaction 정책을 유지할 수 있다.

## 재발 방지 포인트

- provider doctor에 실제 model request, smoke 실행, broker 자동 삭제를 추가하지 않는다.
- `warn`은 실행 전 확인이 필요한 상태이고 `fail`은 사용자의 설치, 인증, 설정 조치가 필요한 상태로 유지한다.
- 새 provider 설정 필드가 추가되면 doctor 전용 parser를 만들지 말고 provider config parser와 계약 문서를 먼저 갱신한다.
- doctor JSON 출력에는 secret 후보, raw stderr, private environment value, raw execution dump를 포함하지 않는다.
- broker state 점검은 PID 생존 여부와 endpoint 접근성 후보를 분리하고, 확정 삭제는 기존 broker lifecycle 정책의 안전 조건을 따른다.
- smoke 전 안내에서는 doctor를 먼저 실행해 환경 문제를 빠르게 분리하고, doctor가 정상이어도 실제 provider 동작은 smoke로 별도 검증한다.
