## 목표

Claude CLI가 process exit 0으로 종료하더라도 terminal `result` 이벤트에 `is_error=true` 또는 `subtype=error`를 담으면 built pipeline이 성공으로 진행되지 않게 한다.
`docs/contracts/provider-events.md`의 Claude stream-json error 매핑과 `runClaude()` 반환값을 맞춰, progress writer와 runner 판단이 같은 failure 상태로 수렴하게 하는 것이 목표였다.

## 구현 내용

- `src/providers/claude.js`의 `runClaude()` close 핸들러에서 terminal result error 상태를 추적하는 `terminalResultIsError` 플래그를 추가했다.
- Claude terminal event가 `result.is_error=true` 또는 `subtype=error`이면 process exit code가 0이어도 `success:false`를 반환하게 했다.
- terminal event에 명시적 `failure` 객체가 있으면 이를 우선 사용하고, 없으면 `model_response` kind와 `claude_result_is_error` code의 failure를 합성한다.
- `test/providers-claude.test.js`에 4개 단위 테스트를 추가했다.
  `is_error=true` 단독, `subtype=error` 단독, 명시적 failure 우선, 정상 result 유지 케이스를 고정했다.

## 결정 사항

- Claude terminal result error는 provider process exit code보다 우선한다.
  Claude stream-json 계약은 `result.is_error`를 error로 매핑하므로, exit 0만 보고 성공 처리하면 문서 계약과 실제 pipeline 판단이 갈라진다.
- 명시적 failure가 없는 terminal result error는 `model_response` failure로 합성한다.
  인증, 설정, sandbox, runner IO 문제가 아니라 provider가 정상 종료했지만 모델 결과가 error terminal로 끝난 상황이기 때문이다.
- `progress-writer.js`와 `runClaude()`는 `is_error`와 `subtype=error`를 모두 terminal failure 신호로 본다.
  writer artifact만 failed가 되고 runner 반환값은 success로 남는 split-brain 상태를 막기 위한 기준이다.

## 재발 방지 포인트

- provider process exit code와 provider terminal event 의미가 충돌하면 terminal event의 failure 신호를 우선 확인한다.
- `progress.json`이 failed로 기록되는 경로와 provider runner 반환값이 서로 다른 성공/실패를 내지 않도록 같은 terminal error predicate를 공유한다.
- Claude terminal `result`에서 `failure` 객체가 누락될 수 있으므로 `is_error`와 `subtype=error`만 있는 케이스를 별도 회귀 테스트로 유지한다.
- `model_response` failure 합성 code를 바꾸면 계약 문서, progress writer 기대값, runner 반환값 테스트를 함께 갱신한다.
