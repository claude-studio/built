## 목표

한 이슈에서 기본적으로 open PR이 하나만 유지되도록 Builder, Reviewer, Finisher의 확인
절차와 merge gate를 보강한다.

## 구현 내용

- `docs/contracts/issue-pr-mapping.md`에 Builder의 PR 생성 전 중복 open PR 조회, 기존 PR 재사용,
  `kg/issues/BUI-<N>.md` mapping 스켈레톤 작성, PR 제목/branch의 BUI 번호 포함 기준을 정리했다.
- `docs/ops/pr-merge-gate.md`에 Finisher G2 중복/stale PR 감지 gate를 추가하고, canonical PR이
  하나가 아니면 merge하지 않고 Coordinator 판단을 요청하도록 했다.
- `kg/agents/builder.md`, `kg/agents/reviewer.md`, `kg/agents/finisher.md`에 같은 BUI 번호의
  open PR 확인, 중복 PR 차단, 한국어/KST 정리 코멘트 작성 기준을 반영했다.
- `scripts/check-pr-merge-ready.js --issue BUI-<N>` 경로와 단위 테스트를 추가해 같은 이슈의
  open PR이 1개가 아니면 merge ready 검증에서 실패하도록 했다.
- `kg/issues/BUI-186.md`에 PR branch mapping 스켈레톤을 만들고, Recorder 단계에서 본 기록과
  durable KG 파일 목록을 완성했다.

## 결정 내용

- built 개발 플로우의 기본 운영 원칙은 "한 이슈 하나의 canonical open PR"이다.
- canonical PR은 `kg/issues/BUI-<N>.md` frontmatter의 `pr` 값과 최신 handoff comment의 PR URL을
  기준으로 판단한다.
- Builder는 PR 생성 전에 `BUI-<N> in:title`와 head branch 기준으로 기존 open PR을 확인한다.
  기존 PR이 있으면 새 PR을 만들지 않고 같은 branch에 추가 commit한다.
- Reviewer와 Finisher는 같은 BUI 번호의 open PR이 여러 개이거나 PR 제목/branch/mapping이
  canonical 판정에 부족하면 PASS 또는 merge를 중단한다.
- 중복 PR 정리 요청 코멘트에는 canonical 후보, 중복 PR 번호, head branch, 다음 조치 요청을
  한국어/KST 기준으로 남긴다.
- PR 제목은 `[BUI-<N>] <한글 요약>`, branch는 `agent/builder/BUI-<N>-<slug>` 형식을 우선한다.

## 결정 이유

- 한 이슈에서 중복 PR이 열리면 Reviewer PASS, Recorder KG 기록, Finisher merge 대상이 갈라져
  같은 변경을 두 번 검토하거나 잘못된 PR을 merge할 위험이 있다.
- 기존 `kg/issues/` mapping SSOT를 활용하면 새 저장소 구조나 외부 DB 변경 없이 canonical PR을
  추적할 수 있다.
- PR 제목과 branch에 BUI 번호를 포함하면 GitHub 검색, stale branch cleanup, handoff comment
  검증이 같은 키로 맞물린다.
- 정상적으로 PR이 1개인 흐름은 추가 확인만 수행하고 기존 Builder->Reviewer->Recorder->Finisher
  순서를 유지한다.

## 재발 방지 포인트

- Builder는 새 PR을 만들기 전에 같은 BUI 번호의 open PR과 같은 head branch open PR을 모두 확인한다.
- 기존 open PR이 있으면 해당 branch에 추가 commit하고, branch 접근이 불가능하면 새 PR을 만들지
  않는다.
- Reviewer PASS와 Finisher merge 전에는 open PR이 정확히 1개인지 확인하고, 여러 개이면 merge
  대상 정리를 먼저 요청한다.
- PR 제목 또는 branch에 BUI 번호가 없으면 canonical 판정이 약해지므로 제목/mapping 정정을 먼저
  요구한다.
- provider comparison처럼 의도적으로 여러 PR이 필요한 실험 흐름은 별도 정책과 식별자를 사용해
  canonical 개발 PR 흐름과 섞지 않는다.
