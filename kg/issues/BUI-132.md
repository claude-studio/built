## 목표

provider가 늘어날 때 실패 원인을 인증, 설정, sandbox, timeout, model 응답, runner normalize, runner IO 등으로 분리한다.
운영자가 `state.json`, `progress.json`, `logs/<phase>.jsonl`만 보고 사용자가 해야 할 조치와 디버그 근거를 구분할 수 있게 한다.

## 구현 내용

- `src/providers/failure.js`를 추가해 모든 provider failure 정규화의 진입점으로 삼았다.
  `FAILURE_KINDS`, `createFailure()`, `classifyClaudeFailure()`, `classifyCodexFailure()`, `failureToEventFields()`, `sanitizeDebugDetail()`을 제공한다.
- `src/providers/codex.js`는 auth, config, sandbox, timeout, broker busy/start 실패, provider unavailable, model response 계열 실패를 표준 `failure` 객체로 emit한다.
- `src/providers/claude.js`는 timeout, spawn 실패, 비정상 stderr, JSON parse 실패를 표준 `failure` 객체로 반환한다.
- `src/providers/event-normalizer.js`는 Claude `result(error)`를 `model_response` failure로 분류하고, Codex error event에 `failure`가 없으면 fallback failure를 보완한다.
- `src/providers/standard-writer.js`는 error event의 `failure`를 `progress.json.last_failure`에 기록하되 `debug_detail`은 제외한다.
- `scripts/run.js`는 phase script 종료 후 `progress.json.last_failure`를 읽어 `state.json.last_failure`로 승격한다.
- `docs/contracts/provider-events.md`와 `docs/contracts/file-contracts.md`에 표준 error event의 `failure` 객체와 `last_failure` 계약을 추가했다.
- `test/providers-failure.test.js`를 추가하고 Codex, Claude, normalizer, file contract 테스트를 새 실패 분류와 맞게 보강했다.

## 결정 사항

- 기존 `message`와 `retryable`은 하위 호환 필드로 유지하고, 구조화된 정보는 새 `failure` 객체에 둔다.
  문자열 필드를 객체로 바꾸면 기존 status/테스트/사용자 표시 경로가 깨질 수 있기 때문이다.
- provider raw error는 사용자 메시지로 직접 노출하지 않고 `failure.debug_detail`에 sanitize 후 보관한다.
  사용자-facing 필드는 `user_message`와 `action`으로 제한해 조치 중심 메시지를 보장하고 secret 노출 위험을 줄인다.
- `progress.json.last_failure`는 최신 실패 snapshot으로 기록하고, `state.json.last_failure`는 runner가 progress에서 단방향 승격한다.
  provider가 lifecycle SSOT인 `state.json`을 직접 쓰지 않는 기존 provider/runner 책임 경계를 유지하기 위한 선택이다.
- `logs/<phase>.jsonl`에는 표준 error event 전체를 남기고, `progress.json`과 `state.json`에는 orchestration과 사용자 조치에 필요한 요약만 남긴다.
- Claude `result(error)`와 JSON parse 실패는 `model_response`, `retryable=true`, `blocked=false`로 분류한다.
  현재 `retryable`은 즉시 자동 재실행 트리거가 아니라 분류 신호이므로 blocked와 구분하는 것이 요구사항에 맞다.

## 재발 방지 포인트

- provider failure kind나 메시지를 추가할 때는 `src/providers/failure.js`의 classifier와 `docs/contracts/provider-events.md`의 kind 목록을 함께 갱신한다.
- raw stderr, provider notification error, token 후보 문자열은 사용자 메시지나 `progress.json`/`state.json`에 직접 넣지 않는다.
  항상 `sanitizeDebugDetail()`을 거쳐 `logs/<phase>.jsonl`의 `failure.debug_detail` 계층에만 둔다.
- `last_error`는 문자열 하위 호환 필드로 유지한다.
  구조화된 판단은 `last_failure`에 추가하고 기존 필드 의미를 바꾸지 않는다.
- `state.json`은 lifecycle SSOT이므로 provider가 직접 쓰지 않는다.
  phase script가 남긴 `progress.json.last_failure`를 runner가 승격하는 단방향 구조를 유지한다.
- `model_response`를 실제 자동 재시도 정책에 연결하려면 retry 횟수와 상한을 별도 계약으로 추가해야 한다.
  현재 `retryable=true`는 분류 신호일 뿐 무제한 재실행 허가가 아니다.
- `provider_unavailable`은 kind 하나로 묶더라도 `code`와 `action`으로 binary 없음, app-server 미지원, broker busy/start 실패를 구분해야 한다.
