## 목표

provider 구조 전환 이후 사용자 문서와 운영 문서에 남아 있던 Claude 전용 표현을 provider 중립 표현으로 정리한다.
또한 built의 provider 선택 축과 Multica agent runtime이 서로 다른 축임을 명확히 하며,
Claude 기본값, Codex opt-in, smoke 분리, usage optional 정책을 문서 기준과 맞춘다.

## 구현 내용

- `README.md`에서 Claude 전용 실행 설명을 provider 서브프로세스 표현으로 정리했다.
  기본 provider는 Claude이고 Codex는 `.built/config.json`의 `providers` 설정으로 명시 opt-in한다는 안내를 추가했다.
- `README.md`에 built provider와 Multica agent runtime은 별개 축이라는 참고 설명을 추가했다.
  예를 들어 Multica에서 Recorder/Builder 같은 agent role로 실행되더라도 built 내부 phase provider는 Claude 또는 Codex 중 하나로 별도 선택된다.
- `README.md`에 provider가 달라도 결과 파일 계약(`progress.json`, `logs`, `do-result.md`, `check-result.md`, `report.md`)은 동일하게 유지된다고 명시했다.
- `BUILT-DESIGN.md`의 실행 모델, 온보딩 가이드, 비교 표에서 Claude 전용 표현을 provider 중립 표현으로 정리했다.
- `docs/roadmaps/provider-transition.md`에서 PR 6 상태를 완료로 갱신하고 산출물을 기록했다.

## 결정 사항

- 사용자-facing 문서에서는 "Claude only"처럼 읽히는 표현을 피하고,
  "provider 서브프로세스(기본값: Claude, opt-in: Codex)"처럼 현재 기본값과 확장 지점을 함께 적는다.
- provider 설정이 없으면 Claude 기본값으로 기존 동작을 유지한다.
  Codex는 phase별 provider 설정으로 명시할 때만 사용한다.
- built provider와 Multica agent runtime은 같은 개념으로 설명하지 않는다.
  Multica agent role은 운영/협업 런타임이고, built provider는 feature phase를 실행하는 로컬 subprocess 선택이다.
- smoke 실행 방식은 기본 테스트와 실제 provider smoke를 분리한다.
  실제 Codex smoke는 인증, 로컬 app-server, 환경 의존성이 있으므로 명시 opt-in 정책을 따른다.
- usage/cost 계열 정보는 provider별로 없을 수 있으므로 optional contract로 유지한다.

## 재발 방지 포인트

- 사용자 문서에서 "Claude subprocess"처럼 특정 provider만 지칭하면 Codex opt-in 경로가 없는 것처럼 보인다.
  기본값을 설명해야 할 때는 "기본값: Claude"와 "opt-in: Codex"를 함께 적는다.
- Multica agent runtime과 built provider를 같은 축으로 섞어 설명하면 운영 역할과 phase 실행 provider의 책임 경계가 흐려진다.
  문서에서는 두 축을 별도 문장으로 분리한다.
- provider 교체 시에도 결과 파일 계약은 provider별 문서로 복제하지 않는다.
  `docs/contracts/provider-events.md`와 runner/writer 계약을 기준으로 provider-neutral 산출물 불변성을 유지한다.
- 실제 provider smoke는 기본 테스트에 넣지 않는다.
  로컬 인증과 app-server 상태에 의존하므로 환경 변수 opt-in으로만 실행한다.
