## 목표

`scripts/iter.js`와 `scripts/report.js`가 `.built/runtime/runs/<feature>/run-request.json`의 `providers.iter`와 `providers.report` 설정을 읽어 phase별 provider opt-in을 지원하게 한다.
설정이 없는 기존 run-request는 Claude 기반 동작을 유지하고, `iter` 루프의 check 재실행과 `report.md` frontmatter 계약이 provider 전환으로 흔들리지 않게 한다.

## 구현 내용

- `scripts/iter.js`가 `parseProviderConfig()`로 provider 설정을 읽고 `providers.iter`를 우선 사용하도록 연결했다.
- `providers.iter`가 없고 `providers.do`가 있으면 `providers.do` spec을 fallback으로 사용하게 했다.
  `iter`는 `do`의 수정 루프이므로 provider와 sandbox 정책이 같은 축에서 이어져야 하기 때문이다.
- `providers.iter`와 `providers.do`가 모두 없으면 기존 Claude provider 기본값을 유지했다.
- `iter` 실행 시 선택된 `providerSpec`을 `runPipeline()`의 `phase: "iter"` 실행에 전달했다.
- `runCheckScript()`는 기존처럼 `scripts/check.js`를 subprocess로 실행한다.
  `check.js`가 같은 `run-request.json`을 직접 읽기 때문에 `providers.check` 설정은 iter 루프 안의 check 재실행에서도 유지된다.
- `scripts/report.js`가 `providers.report` 단축형과 상세형 설정을 읽어 `runPipeline()`에 전달하게 했다.
- `providers.report`가 없으면 기존 Claude + 저비용 기본 모델 흐름을 유지했다.
- `report.md` frontmatter에 `provider`를 추가해 실제 실행 provider와 `model`이 모순되지 않게 정규화했다.
- `test/iter.test.js`에 iter provider routing, do fallback, Codex read-only sandbox 실패, check 재실행 provider 유지 검증을 추가했다.
- `test/report.test.js`에 report provider routing, legacy fallback, frontmatter provider/model 정규화 검증을 추가했다.

## 결정 사항

- `iter` provider 우선순위는 `providers.iter` -> `providers.do` -> Claude 기본값이다.
  `iter`는 독립 phase처럼 호출되지만 실제 의미는 do 산출물을 수정하는 반복 루프라서, 명시적 iter 설정이 없을 때 do provider를 이어받는 것이 provider routing matrix와 가장 잘 맞는다.
- Codex `iter`에는 `workspace-write` sandbox 정책을 유지한다.
  `iter`는 do-result 산출물을 다시 쓰는 phase이므로 check/report처럼 read-only로 완화하면 provider 실행 전제와 실제 산출물 쓰기 요구가 충돌한다.
- `report` provider 기본값은 Claude + `claude-haiku-4-5-20251001` 모델 흐름으로 유지한다.
  report는 최종 요약 산출물이며 기존 사용자의 비용/인증 기대를 바꾸지 않아야 하므로 Codex는 명시적 opt-in으로만 사용한다.
- `report.md`에는 `provider`와 `model`을 실행 메타로 기록한다.
  usage/cost는 optional telemetry이므로 필수 frontmatter 계약으로 승격하지 않고, provider/model/duration 같은 실행 메타와 구분한다.
- iter 내부 check 재실행은 provider 설정을 별도로 복사하지 않는다.
  check subprocess가 동일 run-request를 SSOT로 읽는 구조가 phase별 provider 설정의 중복 전달보다 단순하고 회귀 가능성이 낮다.

## 재발 방지 포인트

- `iter` fallback을 `providers.iter` 부재 시 곧바로 Claude로 되돌리면 Codex do 이후 수정 루프가 provider를 잃는다.
  명시적 iter 설정이 없을 때는 먼저 `providers.do`를 이어받아야 한다.
- `iter`는 check처럼 read-only phase가 아니다.
  Codex read-only 허용을 넓히면 do-result 재작성 phase에서 늦은 실패가 발생하므로 parser 단계에서 빠르게 실패시키는 정책을 유지한다.
- `report` 기본 모델은 top-level `model`과 `providers.report.model`의 우선순위를 혼동하지 않는다.
  `providers.report.model`이 있으면 실제 report provider 실행 모델로 기록되어야 한다.
- iter 내부 check 재실행에 `providers.check`를 별도 인자로 복제하지 않는다.
  `check.js`가 같은 run-request를 읽는 구조를 깨면 설정 SSOT가 둘로 갈라진다.
- `provider`, `model`, `duration_ms`는 실행 메타이고 usage/cost는 optional telemetry다.
  optional usage 값이 없다는 이유로 status/report 계약을 실패 처리하지 않는다.
