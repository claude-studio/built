## 목표

provider 동등성 fixture를 기존 `do` 중심 검증에서 `check`, `iter`, `report`까지 확장해 phase별 파일 계약 회귀를 더 촘촘히 잡는다.
provider가 달라도 필수 frontmatter와 key set이 유지되는지 확인하고, usage 이벤트가 없는 provider와 usage 이벤트가 있는 provider를 모두 offline fake provider로 검증하는 것이 목표였다.

## 구현 내용

- `test/fixtures/provider-common-input.js`에 usage 이벤트가 없는 Codex 표준 이벤트 fixture, iter 이후 승인된 check 결과 fixture, report source data fixture를 추가했다.
- `PROVIDER_INVARIANT_FIELDS['report.md']`를 `id`, `date`, `status`로 정의하고, `PROVIDER_SPECIFIC_FIELDS['report.md']`를 `provider`, `model`로 정의했다.
- `test/e2e/scenarios/05-provider-equivalence-contracts.js`에 iter 사이클 후 `do-result.md`와 `check-result.md` 불변 필드 유지 검증을 추가했다.
- 같은 시나리오에 `report.md` 필수 frontmatter 계약 검증을 추가해 Claude/Codex provider 간 불변 필드와 provider-specific 필드를 분리해 확인했다.
- 실제 Claude/Codex 호출 없이 `standard-writer`, `writeCheckResult`, `writeReport` helper와 fake fixture만으로 검증되도록 유지했다.

## 결정 사항

- check/iter/report 동등성은 기존 provider equivalence E2E 시나리오 5에 확장했다.
  phase별 산출물 계약을 한 파일에서 비교하면 plan_synthesis, do, check, iter, report 범위를 한 번에 회귀 확인할 수 있기 때문이다.
- usage/cost는 필수 계약으로 승격하지 않고 optional provider telemetry로 유지했다.
  provider에 따라 usage 이벤트가 없을 수 있으므로 파일 계약 동등성은 필수 frontmatter와 key set 중심으로 검증해야 한다.
- report의 `provider`와 `model`은 provider-specific 필드로 남기고, `id`, `date`, `status`만 provider-invariant로 비교했다.
  report 실행 provider와 model은 실제 실행 메타이므로 provider 간 동일해야 하는 값이 아니기 때문이다.

## 재발 방지 포인트

- provider equivalence fixture는 provider 출력 품질 비교가 아니라 파일 계약 회귀 검증이다.
  real Claude/Codex 품질 비교나 비교 모드 구현을 이 시나리오에 섞지 않는다.
- usage/cost가 없는 provider를 실패로 보지 않는다.
  optional telemetry 부재와 필수 산출물 계약 위반을 분리해서 검증한다.
- `report.md`의 `provider`, `model`은 실행 메타라 provider별로 달라야 정상이다.
  동등성 비교에 포함하면 cross-provider 검증이 잘못 실패할 수 있다.
- check/iter/report 산출물 계약을 바꿀 때는 `docs/contracts/file-contracts.md`, `test/fixtures/provider-common-input.js`, `test/e2e/scenarios/05-provider-equivalence-contracts.js`를 함께 확인한다.
