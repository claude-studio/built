## 목표

provider별 지원 phase, sandbox 필요 여부, outputSchema 지원 여부, app-server 필요 여부, 기본 timeout 정책을 `src/providers/capabilities.js` 한 모듈에서 조회하게 한다.
provider 정책이 config parser, 문서, adapter에 흩어져 새 provider 추가 시 drift가 생기는 문제를 줄이는 것이 핵심 목표다.

## 구현 내용

- `src/providers/capabilities.js`를 추가해 Claude/Codex capability registry를 만들었다.
- registry에는 `supportedPhases`, `requiresAppServer`, `supportsOutputSchema`, `defaultTimeoutMs`, `defaultSandbox`, `writeRequiredSandbox`를 provider별로 명시했다.
- 공개 API로 `getCapability`, `isPhaseSupported`, `requiresWrite`, `getDefaultSandbox`, `validateSandbox`를 제공했다.
- `src/providers/config.js`의 `VALID_PROVIDERS`와 sandbox 검증을 capability registry에서 파생하도록 연결했다.
- 알 수 없는 provider 오류는 `capabilities.js` 등록을 안내하는 한글 사용자 조치 중심 메시지를 유지했다.
- `test/providers-capabilities.test.js`를 추가해 Claude/Codex capability 정의, phase 지원, 기본 sandbox, Codex write phase sandbox 정책을 검증했다.

## 결정 사항

- provider capability registry를 provider 지원 범위의 SSOT로 둔다.
  config parser와 doctor/문서는 provider 이름, phase 지원, sandbox 기본값을 직접 중복 정의하지 말고 registry에서 파생해야 한다.
- Claude는 sandbox 개념이 없는 provider로 기록한다.
  `defaultSandbox`와 `writeRequiredSandbox`를 `null`로 두어 기존 Claude CLI 동작과 하위 호환성을 유지한다.
- Codex는 app-server가 필요한 provider로 기록하고, 기본 sandbox는 `read-only`로 둔다.
  `do`와 `iter`처럼 파일 변경이 필요한 phase에서는 `workspace-write`를 요구한다.
- `check`, `report`, `plan_synthesis`는 읽기 중심 phase로 남겨 Codex `read-only`를 허용한다.
- 기본 provider는 변경하지 않는다.
  이번 작업은 capability registry 도입이며 Codex를 기본 provider로 승격하는 정책 변경은 포함하지 않는다.

## 재발 방지 포인트

- 새 provider를 추가할 때 `config.js`나 doctor에 provider 이름을 먼저 추가하지 않는다.
  `src/providers/capabilities.js` 등록을 먼저 하고, 나머지 소비자는 registry에서 파생해야 한다.
- Codex `do`/`iter`를 `read-only`로 허용하지 않는다.
  파일 변경 phase에서 sandbox 권한 부족이 늦게 드러나면 실패 원인과 사용자 조치가 흐려진다.
- Claude에 Codex식 sandbox 기본값을 부여하지 않는다.
  Claude CLI는 현재 sandbox 개념이 없는 provider로 모델링되어야 기존 동작과 충돌하지 않는다.
- registry는 실행 결과 파일을 쓰거나 provider subprocess를 호출하지 않는다.
  capability 조회와 설정 검증 경계만 담당해야 provider 파일 직접 쓰기 금지 원칙과 맞는다.
- 문서나 doctor가 provider capability 표를 추가할 때 같은 정책을 수기로 복제하지 않는다.
  registry에서 파생하거나 registry를 기준으로 동기화되었는지 테스트해야 한다.
