## 목표

README의 Next Step인 execution worktree 재사용 구조를 MVP로 연결해 provider 실행과 원본 레포 상태를 분리한다.
`/built:run` 시작 시 feature 전용 execution worktree를 생성하거나 재사용하고, Do/Check/Iter/Report 및 plan synthesis가 같은 worktree 컨텍스트에서 실행되게 한다.

## 구현 내용

- `scripts/run.js`가 git project에서 execution worktree를 준비하고 `built/worktree/<feature>` branch, worktree path, canonical resultDir, cleanup command를 state/registry에 기록하게 했다.
- Do/Check/Iter/Report와 plan synthesis 호출에 worktree CWD/result output path를 전달해 phase 산출물을 worktree `.built/features/<feature>` 아래에 쌓게 했다.
- root `.built/runtime`은 state, run-request, registry pointer를 보관하는 control-plane 위치로 유지했다.
- legacy escape hatch와 root fallback을 유지해 non-worktree 흐름과 기존 Claude 기본 run이 계속 동작하게 했다.
- `scripts/status.js`와 `scripts/cost.js`가 registry/state의 `resultDir` pointer를 우선 읽고, 없을 때 root `.built/features/<feature>`로 폴백하게 했다.
- `scripts/cleanup.js`가 explicit worktree path 삭제 전 허용 루트, expected branch, dirty 상태를 검증하고 unsafe fallback 삭제를 하지 않게 했다.
- offline 테스트에 execution worktree pointer, canonical 비용 guard, status/cost pointer 소비, cleanup safety gate를 추가했다.

## 결정 내용

- `/built:run` 기본 실행은 worktree-first로 둔다. 단, `BUILT_DISABLE_WORKTREE`와 run-request `execution_worktree=false` 같은 legacy escape hatch를 유지한다.
- root `.built/runtime`은 control-plane state/pointer 위치이고, worktree `.built/features/<feature>`는 phase 산출물과 `progress.json`의 canonical 위치다.
- status/cost/cleanup은 root feature dir을 추측하기 전에 registry/state pointer를 먼저 소비해야 한다.
- cleanup은 registry/state의 explicit worktree path가 있어도 허용 루트, expected branch, dirty 상태 검증을 통과한 경우에만 삭제한다.

## 결정 이유

- provider 실행이 원본 레포 working tree를 직접 오염시키지 않도록 phase 실행 컨텍스트와 결과 위치를 분리해야 했다.
- root runtime을 유지하면 기존 state machine, run-request handoff, registry 소비자를 크게 바꾸지 않고 worktree-first MVP를 도입할 수 있다.
- pointer를 기준으로 소비자를 정렬하지 않으면 `/built:status`, `/built:cost`, 비용 guard가 worktree canonical 결과가 아니라 오래된 root fallback을 볼 수 있다.
- cleanup은 경로 삭제를 포함하므로 unsafe path, branch mismatch, dirty worktree를 발견하면 runtime/result cleanup까지 중단하는 보수적 정책이 필요했다.

## 재발 방지 포인트

- execution worktree 관련 변경은 run 성공만 보지 말고 비용 guard, status, cost, cleanup이 모두 같은 `resultDir` pointer를 소비하는지 확인한다.

- provider adapter는 산출물 파일을 직접 쓰지 않고 runner/writer normalization 경계를 유지한다.
- worktree-first 기본 동작을 강화하더라도 legacy/non-worktree fallback 테스트를 함께 유지한다.
