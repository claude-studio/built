## 목표

provider 전환, 운영 안정화, 품질 최적화, 제품화 backlog가 완료된 뒤 사용자-facing README를 실제 구현 기준으로 재정렬한다.
README만 보고 Claude 기본 실행, Codex phase별 opt-in, smoke 실행, 실패 대응, provider 설정 위치, usage/cost optional telemetry 정책을 이해할 수 있게 한다.

## 구현 내용

- `README.md`를 전면 재구성 수준으로 업데이트해 provider 전환 이후 최종 동작 방향을 반영했다.
- Claude 기본값과 Codex opt-in 실행의 차이를 사용자 관점에서 설명했다.
- `plan_synthesis`와 `do` phase의 provider 선택 예시를 실제 `run-request.json`의 `providers` 필드와 parser 동작에 맞췄다.
- `npm run test:smoke:codex` 전체 실행 명령과 smoke/fake 테스트 분리를 README에서 찾을 수 있게 했다.
- provider failure taxonomy와 Claude/Codex 오류 메시지 대응표를 추가해 인증, 설정, sandbox, timeout, provider unavailable, model response 계열 실패를 조치 중심으로 안내했다.
- worktree/branch cleanup 정책, hooks 사용 가능 상태, usage/cost optional telemetry 정책, provider setup guide와 contract 문서 링크를 보강했다.
- 내부 roadmap, 위험 요소, Next Step, MVP 후순위, 예상 위치 같은 미완료 계획 표현을 제거했다.

## 결정 사항

- README는 provider 전환 완료 후 사용자 onboarding의 최종 진입점으로 유지한다.
  세부 contract 전문은 `docs/contracts/`와 `docs/ops/` 문서에 두고, README는 사용자가 실행과 문제 해결을 시작하는 데 필요한 경로와 판단 기준을 연결한다.
- failure taxonomy는 내부 구현 세부사항이 아니라 사용자-facing troubleshooting의 핵심 분류로 노출한다.
  raw provider stderr가 아니라 `failure.kind`, 사용자 조치, 관련 문서 링크를 기준으로 안내해야 secret 노출 위험 없이 해결 경로를 줄 수 있기 때문이다.
- `plan_synthesis`는 opt-in phase지만 pipeline overview에 명시한다.
  Codex provider 실험과 품질 검토 흐름에서 사용자가 어디에서 phase를 켜는지 알아야 하므로 숨겨진 고급 기능처럼 두지 않는다.
- hooks는 예정 기능이 아니라 현재 사용 가능한 기능으로 기술한다.
  `skills/hooks-inspect`가 실제 존재하므로 MVP 후순위/예상 위치 표현을 남기면 README가 현재 상태와 충돌한다.
- built provider와 Multica agent runtime은 별도 축으로 설명한다.
  Multica 역할 협업 흐름과 built 내부 provider subprocess 선택을 섞으면 설정 책임과 실행 책임이 불명확해진다.
- usage/cost는 optional telemetry로 설명하고 provider/model/duration 같은 실행 메타와 구분한다.
  provider별로 usage/cost 값이 없을 수 있으므로 README에서 필수 실행 계약처럼 보이면 안 된다.

## 재발 방지 포인트

- README의 provider 설정 예시는 `.built/config.json`이 아니라 `.built/runtime/runs/<feature>/run-request.json`의 `providers` 필드를 기준으로 작성한다.
- Codex는 전체 기본값 전환이 아니라 phase별 opt-in이다.
  필요한 phase만 지정하는 예시를 우선하고, 파일을 쓰는 `do`/`iter` 예시에는 `sandbox: "workspace-write"`를 포함한다.
- `providers.plan_synthesis` 설정은 `plan_synthesis` phase를 활성화한다.
  `plan_synthesis: true`를 필수처럼 설명하지 않는다.
- failure troubleshooting은 raw stderr 전문이 아니라 failure kind, 사용자 조치, contract/ops 문서 링크 중심으로 작성한다.
- usage/cost는 optional telemetry다.
  provider/model/duration 같은 실행 메타와 섞어 필수 산출물처럼 설명하지 않는다.
- 향후 execution worktree 구조가 사용자-facing 디렉터리 구조에 노출될 만큼 안정화되면 README의 디렉터리 구조 섹션을 갱신한다.
