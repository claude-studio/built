## 목표

pre/post hook이 provider phase, model, 완료 상태, 실패 요약을 안전하게 받아 사용자 프로젝트 자동화에 연결할 수 있게 한다.
동시에 hook 실패가 provider run 완료 상태에 미치는 영향을 문서화하고, provider 인증 정보나 외부 서비스 secret이 hook 프로세스로 암묵 전달되지 않게 한다.

## 구현 내용

- `src/hooks-runner.js`에 `providerContext` 옵션을 추가해 hook 프로세스 환경변수로 `BUILT_PROVIDER`, `BUILT_PHASE`, `BUILT_PROVIDER_STATUS`, `BUILT_FAILURE_SUMMARY`, `BUILT_MODEL`을 주입한다.
- 기존 hook 호출자가 `providerContext`를 넘기지 않는 경우 위 provider-aware 환경변수는 빈 문자열로 주입해 하위 호환을 유지한다.
- `buildHookEnv()`와 `SENSITIVE_ENV_PATTERN`을 추가해 `_KEY`, `_SECRET`, `_TOKEN`, `_PASSWORD`, `_CREDENTIAL`, `_PRIVATE_KEY`, `_CLIENT_SECRET`, `_AUTH_TOKEN`, `_REFRESH_TOKEN`, `_ACCESS_TOKEN` 접미어 환경변수를 hook payload에서 제외한다.
- `docs/contracts/provider-events.md`에 Hook Payload 정책을 추가해 provider-aware 환경변수 계약, condition 표현식 범위, 민감정보 제외 기준, hook 실패 영향 테이블을 문서화했다.
- `scripts/hooks-inspect.js`가 `before_check`, `before_report`를 포함한 hook point와 provider-aware context 힌트, 민감정보 미전달 안내를 출력하도록 갱신했다.
- `test/hooks-runner.test.js`에 provider-aware context 전달과 민감정보 필터링 회귀 테스트를 추가했다.

## 결정 내용

- provider-aware hook context는 provider 파일 계약을 직접 쓰게 하지 않고 hook 프로세스 환경변수로만 노출한다.
  hook은 사용자 정의 임의 프로세스이므로 provider event/state writer 계층과 결합시키지 않는 것이 안전하다.
- `condition` 표현식에는 provider context를 추가하지 않고 기존 `feature.*`, `check.*` 경로만 유지한다.
  hook 실행 조건 언어를 확장하면 expression evaluator 계약과 보안 표면이 함께 커지므로 이번 범위에서는 환경변수 접근으로 제한했다.
- hook 실패는 provider run terminal event를 재작성하지 않는다.
  `halt_on_fail`과 hook point 조합에 따라 pipeline 진행 여부, check result 강제 변경 여부, report 진입 여부를 결정하고 그 정책을 문서와 테스트로 고정한다.
- hook 환경은 `process.env` 전체 복사가 아니라 민감 접미어를 제거한 환경 위에 built 전용 컨텍스트를 주입한다.
  외부 서비스 연동이 필요한 hook은 별도 hook 설정이나 hook 전용 env 파일을 통해 명시적으로 secret을 읽어야 한다.

## 재발 방지 포인트

- hook payload에 provider context를 추가할 때 provider raw event, token, private environment value, provider 파일 writer 입력을 그대로 넘기지 않는다.
- hook condition 표현식에 provider context 경로를 추가하려면 evaluator 보안 표면과 하위 호환을 별도 이슈에서 검토한다.
- `BUILT_FAILURE_SUMMARY`는 실패 요약용 문자열이며 raw stderr나 secret 후보를 싣는 필드가 아니다.
- hook이 외부 서비스 secret을 필요로 하면 `process.env` 암묵 전파에 기대지 말고 hook 전용 설정으로 명시 주입한다.
- hook 실패 정책을 바꿀 때는 `halt_on_fail`, hook point, provider phase 종료 상태, check/report 산출물 영향이 서로 뒤집히지 않도록 문서 테이블과 테스트를 함께 갱신한다.
