## 목표

Claude headless Do 단계에서 `tool_result` 또는 `user` 이벤트가 `This command requires approval` 계열 승인 거부를 반복할 때 무한 대기처럼 진행되지 않게 한다.
반복 approval denial은 자동 승인하거나 일반 model output으로 방치하지 않고, `claude_permission_request` 계열 blocked failure로 종료해 runner가 `state.json`과 lock을 정리하게 하는 것이 목표였다.

## 구현 내용

- `src/providers/claude.js`가 Claude stream-json의 `tool_result`와 `user` 이벤트에서 approval denial 문구를 감지한다.
- 같은 approval denial이 반복되면 synthetic terminal `result(error)`를 writer 경로에 전달하고 provider 결과를 `success:false`로 반환한다.
- timeout 또는 approval loop로 provider를 종료할 때 process group `SIGTERM`을 먼저 시도하고, 실패하면 기존 child kill fallback을 사용한다.
- `src/providers/failure.js`의 Claude permission request detector에 `This command requires approval` 문구를 추가했다.
- `test/providers-claude.test.js`, `test/providers-failure.test.js`, `test/pipeline-runner.test.js`에 반복 tool approval denial 회귀 케이스를 추가했다.

## 결정 사항

- 반복 tool approval denial은 기존 `failure.kind=model_response`, `failure.code=claude_permission_request`, `blocked=true`, `retryable=false` 분류를 사용한다.
  headless Claude 실행에서 interactive approval을 처리할 수 없고, 같은 환경에서 같은 명령을 재시도하면 산출물 없이 같은 denial만 반복되기 때문이다.
- built는 npm create 같은 명령을 자동 허용하지 않는다.
  provider permission 정책을 우회하면 사용자의 보안 경계를 깨고, 이번 결함의 본질인 "승인 불가 상태를 실패로 수렴하지 못함"을 숨기게 된다.
- provider는 실패 이벤트와 결과만 반환하고 `state.json` 또는 lock을 직접 정리하지 않는다.
  runner/control plane이 lifecycle SSOT를 유지해야 하므로, failure를 `success:false`로 돌려 기존 finally 경로가 `state.json.status=failed`와 lock release를 처리하게 한다.
- orphan `claude -p` 방지는 provider child process뿐 아니라 process group 종료를 시도한다.
  Claude CLI가 하위 프로세스를 가질 수 있어 단일 child kill만으로는 stale provider process가 남을 수 있기 때문이다.

## 재발 방지 포인트

- Claude stream 중간 이벤트의 `tool_result`/`user` approval denial은 terminal result에 도달하지 않을 수 있다.
  terminal `result(success)` 감지만으로는 BUI-268과 다른 approval loop를 막을 수 없다.
- `claude_permission_request` 계열 failure는 자동 승인 대상이 아니라 사용자의 provider permission 설정 변경이 필요한 blocked 상태다.
- provider가 failure를 직접 파일에 쓰거나 lock을 만지지 않게 한다.
  writer와 runner cleanup 경로로 수렴해야 progress/state/lock 계약이 갈라지지 않는다.
- Claude provider 종료 로직을 바꿀 때는 child process group 종료와 fallback kill을 함께 검증한다.
  단일 child kill로 충분하다고 가정하면 orphan provider process와 stale lock이 재발할 수 있다.
