## 목표

user-facing Codex runner 계약을 `/built:run-codex`로 정리한다.
이 명령은 일반 run의 Do/Check/Iter/Report 4단계를 모두 Codex로 실행해야 하며,
`plan_synthesis`까지 실행하는 advanced/internal preset과 섞이면 안 된다.

## 구현 내용

- `src/providers/presets.js`에 `codex-run` preset을 추가했다.
  이 preset은 `do`, `check`, `iter`, `report`를 Codex로 라우팅하고 `plan_synthesis`는 포함하지 않는다.
- `skills/run-codex-do/SKILL.md`를 user-facing command에서 제거하고 `skills/run-codex/SKILL.md`로 교체했다.
- `/built:run-codex <feature>`는 `provider-preset.js <feature> --preset codex-run`으로 run-request를 만든 뒤 `run.js <feature>`를 실행하도록 문서화했다.
- README, `skills/run`, provider failure/remediation 문구, status 안내, packaging/preset 테스트를 새 명령 계약에 맞췄다.
- `kg/workflows/plugin-packaging-validation.md`와 `kg/workflows/provider-preset-helper-validation.md`의 검증 대상도 `run-codex` 기준으로 갱신했다.

## 결정 사항

- `/built:run-codex`는 Do/Check/Iter/Report 전부 Codex를 뜻하는 최종 user-facing runner 명령으로 둔다.
  `run-codex-do`는 BUI-305에서 임시로 노출된 mixed preset 명령이었고, primary command로 유지하지 않는다.
- `codex-run`은 일반 run 4단계 전용 preset으로 둔다.
  일반 runner 명령이 Plan synthesis까지 실행하면 Do -> Check -> Iter -> Report라는 사용자 기대와 다르기 때문이다.
- `codex-all`은 `plan_synthesis`까지 포함하는 advanced/internal preset으로 유지한다.
  기존 내부 실험과 plan synthesis smoke 경로를 깨지 않으면서 user-facing runner만 명확히 분리하기 위한 선택이다.
- `codex-do`는 내부/advanced preset으로 남긴다.
  기존 mixed routing 테스트와 과거 산출물 호환을 보존하되, 새 primary command로 노출하지 않는다.

## 발생한 이슈

- BUI-305에서 `/built:run-codex-do`가 먼저 머지됐으나 runner UX 논의 후 최종 명령 세트와 맞지 않는 것으로 정리됐다.
- `codex-all`은 모든 phase라는 이름과 달리 일반 run 4단계뿐 아니라 `plan_synthesis`도 포함한다.
  따라서 `/built:run-codex`를 단순히 `codex-all`에 연결하지 않고 `codex-run`을 별도 추가했다.

## 재발 방지 포인트

- user-facing runner 명령은 일반 run 단계와 advanced/internal phase를 분리해야 한다.
