## 목표

`scripts/check.js`가 `.built/runtime/runs/<feature>/run-request.json`의 `providers.check` 설정을 읽어 check phase provider를 선택하게 한다.
설정이 없을 때는 기존 Claude check 동작을 유지하고, Codex opt-in 시에도 `check-result.md` 계약과 사용자 조치 중심 실패 표현을 유지한다.

## 구현 내용

- `scripts/check.js`에 `parseProviderConfig()`와 `getProviderForPhase()`를 연결해 `providers.check` 기반 providerSpec을 선택하도록 했다.
- `providers.check`가 없거나 `run-request.json`이 없으면 기본 providerSpec을 `{ name: "claude" }`로 유지했다.
- `providers.check: "codex"` 단축형과 `{ name: "codex", model, effort, sandbox, timeout_ms }` 상세형 설정이 check phase의 `runPipeline()`으로 전달되게 했다.
- provider 상세 설정의 `model`이 있으면 기존 top-level `model`보다 check provider 실행 모델로 사용되게 했다.
- `check-result.md` frontmatter에 `provider`, `model`, `duration_ms` 실행 메타를 추가했다.
- `test/check.test.js`에 provider routing, read-only sandbox 허용, 잘못된 provider 오류, Claude/Codex providerSpec 전달 회귀 테스트를 추가했다.

## 결정 사항

- check phase 기본값은 Claude로 유지한다.
  Codex 인증 또는 app-server 준비가 되지 않은 환경에서도 기존 사용자의 기본 check가 실패하지 않아야 하기 때문이다.
- Codex check는 명시적 `providers.check` 설정으로만 활성화한다.
  ADR-11의 cross-provider review 원칙은 opt-in 정책이며, Codex를 기본 check provider로 승격하는 결정은 별도 안정화 이후에만 가능하다.
- check phase에서 `read-only` sandbox를 허용한다.
  check는 feature 파일을 수정하지 않고 provider 결과를 runner/writer 경로로 정규화하므로, `do`/`iter`의 `workspace-write` 필수 정책과 다르다.
- `scripts/check.js`는 기존 `jsonSchema` 인자를 계속 전달한다.
  `runPipeline()`이 provider별로 Claude `jsonSchema`와 Codex `outputSchema` 차이를 흡수하므로 phase script가 provider별 schema API를 직접 분기하지 않아도 된다.
- provider는 `check-result.md`를 직접 쓰지 않는다.
  provider raw output은 `runPipeline()`에서 구조화 결과로 normalize되고, phase script가 기존 frontmatter와 본문 계약에 맞춰 최종 산출물을 작성한다.

## 재발 방지 포인트

- check phase provider를 바꿀 때도 기본 provider를 암묵적으로 Codex로 바꾸지 않는다.
  기본값 변경은 Codex 인증, app-server, sandbox 준비 상태가 없는 사용자에게 회귀가 된다.
- `scripts/check.js`에서 Claude와 Codex schema API를 직접 분기하지 않는다.
  provider별 `jsonSchema`/`outputSchema` 차이는 `runPipeline()` 경계에 남겨 phase script가 산출물 계약만 관리하게 한다.
- check phase는 read-only sandbox를 허용하지만, `do`/`iter`의 Codex `workspace-write` 필수 정책까지 느슨하게 해석하지 않는다.
- `provider`, `model`, `duration_ms`는 실행 메타다.
  기존 status 판정이나 acceptance criteria 본문 구조를 provider별로 바꾸는 데 사용하지 않는다.
- cross-provider review는 built 내부 provider 설정과 Multica Reviewer 역할을 분리해서 설명한다.
  `providers.check: "codex"`는 built check phase 실행 provider 선택이며, PR 리뷰 담당 agent 변경이 아니다.
