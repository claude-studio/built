## 목표

standalone `do`, `check`, `iter`, `report`, `plan-synthesis` phase가 malformed `run-request.json`을 `null` request처럼 삼켜 기본 provider로 진행하지 않게 한다.
`/built:run`과 standalone phase의 run-request parse failure, missing request fallback, provider routing 계약을 같은 기준으로 맞춘다.

## 구현 내용

- `src/run-request.js`에 공통 run-request reader, parse failure error, provider routing helper, 오류 출력 helper를 추가했다.
- `scripts/run.js`와 standalone `scripts/do.js`, `scripts/check.js`, `scripts/iter.js`, `scripts/report.js`, `scripts/plan-synthesis.js`가 malformed `run-request.json`을 같은 계열 메시지와 exit code 1로 실패하도록 정렬했다.
- standalone phase가 `run-request.providers`를 우선 사용하고, 없으면 `.built/config.json.default_run_profile`, 그것도 없으면 built default provider를 쓰도록 `/built:run` fallback과 맞췄다.
- `report`는 기존 저비용 기본 모델 정책을 유지하기 위해 provider가 명시된 경우에만 provider override를 적용한다.
- `test/standalone-run-request.test.js`를 추가해 대상 standalone phase의 malformed request hard fail과 missing request 허용 fallback을 고정했다.
- `test/check.test.js`, `test/report.test.js`, 기존 run/provider 테스트를 보강해 provider 선택 계약 회귀를 막았다.

## 결정 내용

- malformed `run-request.json`은 모든 phase에서 실행 snapshot 오류로 간주하며, silent fallback 대상이 아니다.
- `run-request.json`이 아예 없는 경우는 기존 standalone 허용 범위에 맞춰 fallback할 수 있지만, malformed JSON과는 명확히 구분한다.
- provider routing 우선순위는 `run-request.providers` → `.built/config.json.default_run_profile` → built default이다.
- standalone phase도 provider 설정 오류를 phase-local 기본값으로 숨기지 않고 config path와 원인을 보여준 뒤 실패한다.

## 결정 이유

- malformed request가 기본 provider fallback으로 처리되면 사용자는 provider routing, `planPath`, 비용 제한, `plan_synthesis` 옵션이 무시된 실행을 성공으로 오해할 수 있다.
- `/built:run`과 직접 phase 실행의 계약이 다르면 같은 feature가 실행 경로에 따라 다른 provider와 옵션으로 처리된다.
- missing request는 legacy standalone 사용성과 관련된 경로이지만, malformed request는 사용자가 만든 실행 snapshot의 무결성 오류이므로 복구 기준이 다르다.
