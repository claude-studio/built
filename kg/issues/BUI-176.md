## 목표

BUI-168에서 추가한 provider doctor diagnostics를 Claude Code plugin 사용자 경험에 연결한다.
사용자는 `/built:doctor` 명령으로 Codex CLI 설치, app-server 지원, 인증 상태, broker 상태, run-request provider 설정을 실제 provider 실행 없이 사전 점검할 수 있어야 한다.

## 구현 내용

- `skills/doctor/SKILL.md`를 추가해 `/built:doctor [--feature <featureId>] [--json]` 사용자 진입점을 정의했다.
- plugin 설치 경로에서는 skill 파일 기준 `../../scripts/provider-doctor.js`를 찾아 실행하도록 안내했다.
- 로컬 개발과 `--plugin-dir` 경로에서는 `node scripts/provider-doctor.js --cwd "$(pwd)"`를 직접 실행하는 방법을 함께 문서화했다.
- skill 출력 예시는 한글 사용자 메시지를 기준으로 유지하고, 실패 항목에는 조치 메시지가 함께 표시된다는 점을 명시했다.
- skill 비범위에 실제 모델 호출, token/secret/env 원문 출력, GUI 설정 화면을 제외한다고 명시했다.
- `README.md` 유틸리티 명령 목록에 `/built:doctor`를 추가했다.
- `docs/ops/provider-setup-guide.md`의 provider doctor 섹션에 plugin 사용 명령과 직접 실행 명령을 나누어 연결했다.

## 결정 내용

- `/built:doctor` skill은 diagnostics 로직을 새로 구현하지 않고 `scripts/provider-doctor.js`를 실행하는 사용자 UX 진입점으로 둔다.
- 실제 점검 책임은 BUI-168에서 추가된 `src/providers/doctor.js`와 `scripts/provider-doctor.js`에 유지한다.
- skill은 경로 해석, 사용자 인자 전달, 사용법 문서화, 비범위 안내에 집중한다.
- provider doctor는 사전 점검 전용이며 실제 모델 호출이나 smoke 실행을 수행하지 않는다.

## 결정 이유

- skill과 diagnostics script가 같은 점검 규칙을 중복 구현하면 Codex 설치, 인증, broker, sandbox/config 오류 해석이 drift될 수 있다.
- plugin 사용자와 로컬 개발자가 같은 script를 호출해야 완료 기준과 테스트 기준을 하나로 유지할 수 있다.
- skill이 raw env나 provider stderr를 직접 다루지 않으면 ADR-9의 사용자 메시지와 debug detail 분리 원칙을 유지하기 쉽다.
- `/built:doctor`를 README와 provider setup guide에 연결해야 provider 전환 사용자가 smoke 전에 점검 진입점을 발견할 수 있다.

## 재발 방지 포인트

- `/built:doctor` skill에는 provider 점검 로직을 중복 구현하지 않는다.
- provider doctor 점검 항목이 바뀌면 `scripts/provider-doctor.js`, `src/providers/doctor.js`, `skills/doctor/SKILL.md`, `docs/ops/provider-setup-guide.md`를 함께 검토한다.
- skill 예시에는 secret, token, private environment value, raw execution dump를 포함하지 않는다.
- `--feature`가 필요한 run-request provider 설정 검증은 기본 점검과 구분해 문서화한다.
- doctor가 정상이어도 실제 provider 응답과 산출물 계약은 smoke 테스트에서 별도로 검증한다.
