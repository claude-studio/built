## 목표

긴 provider 실행에서 `progress.json`과 `scripts/status.js` 출력이 대용량 `tool_result`, 최종 result, error message를 반복 포함해 agent context와 status 화면을 과도하게 소비하지 않도록 한다.
사용자-facing snapshot은 요약과 최근 tail만 보이게 하되, 원본 audit event는 `.built/features/<feature>/logs/<phase>.jsonl`에 손실 없이 보존해야 했다.

## 구현 내용

- `src/progress-compaction.js`를 추가해 progress/log 표시용 compaction 기준을 한 곳에 모았다.
- `src/progress-writer.js`와 `src/providers/standard-writer.js`가 표준 event를 관찰하면서 `log_summary`와 `recent_events`를 갱신하도록 했다.
- `progress.json.result`, `result_summary`, `last_error`, `recent_events[].summary`는 bounded string으로 제한하고, 대용량 원본은 JSONL audit log에 남기도록 경계를 정리했다.
- `scripts/status.js`는 전체 로그를 다시 읽는 대신 `progress.json`의 `log_summary`와 `recent_events`를 표시한다.
- `docs/contracts/file-contracts.md`에 progress snapshot compaction과 JSONL 원본 보존 계약을 명시했다.
- `test/progress-writer.test.js`, `test/standard-writer.test.js`, `test/status.test.js`에 result, `tool_result`, `last_error`, status 출력 compaction 회귀 테스트를 추가했다.

## 결정 내용

- `progress.json`은 append-only log가 아니라 현재 상태를 보여주는 compact snapshot으로 유지한다.
- 원본 provider event 전문은 `logs/<phase>.jsonl`의 audit 계층에 보존하고, status/agent context는 기본적으로 `progress.json`의 summary와 recent tail만 읽는다.
- compaction은 표시 계층 정책이며 원본 JSONL 이벤트를 삭제하거나 축약하지 않는다.
- `last_error`는 하위 호환 문자열 필드로 두되, 사용자-facing snapshot이므로 200자 경계를 적용한다.

## 결정 이유

긴 provider 실행에서 `tool_result`나 error 전문이 `progress.json`과 status 출력에 반복 저장되면 status 확인 자체가 비싸지고, agent가 매번 전체 로그를 읽게 된다.
반대로 원본 JSONL을 축약하면 감사와 디버깅 근거가 사라진다.
따라서 compact snapshot과 audit log를 분리해 운영 효율과 추적 가능성을 함께 유지한다.

## 재발 방지 포인트

- 사용자-facing `progress.json`에 대용량 provider 원문을 새로 추가하지 않는다.
  최종 result는 1200자, `tool_result` summary는 500자, error/`last_error`는 200자 경계를 유지한다.
- `recent_events`는 최근 5개 event tail만 담고, 전체 event count와 type별 집계는 `log_summary`에 둔다.
- 감사/디버깅에 필요한 원본 event 전문은 `logs/<phase>.jsonl`에서 확인한다.
  status나 agent handoff가 JSONL 전체를 기본 입력으로 삼지 않게 한다.
- provider adapter는 결과 파일과 progress 파일을 직접 쓰지 않는다.
  compaction 정책은 writer/status 계층에서 유지한다.
- error/failure 경로를 바꿀 때는 success result뿐 아니라 fallback `event.result`, `user_message`, raw message가 모두 bounded snapshot으로 들어가는지 테스트한다.
