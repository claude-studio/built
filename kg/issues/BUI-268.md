## 목표

headless Claude Do 단계가 파일 쓰기 권한 승인 대기 메시지를 반환했는데도 `do-result.md`를 `status: completed`로 기록하는 false-positive 완료를 막는다.
실제 산출물이 생성되지 않은 상태가 Check/Iter 반복으로 뒤늦게 드러나는 대신, provider/model response 실패로 즉시 분류되게 하는 것이 목표였다.

## 구현 내용

- `src/providers/failure.js`에 `isClaudePermissionRequest()`와 `classifyClaudePermissionRequest()`를 추가했다.
  권한, 승인, permission, approval, allowed 같은 영문/한글 키워드 조합으로 Claude 권한 승인 대기 응답을 감지하고 `model_response` kind, `claude_permission_request` code로 정규화한다.
- `src/providers/claude.js`는 Claude `result(success)` 이벤트 본문이 권한 승인 대기 응답이면 `error` 이벤트로 바꾼다.
  이 과정에서 `_dispatchLine` 반환값은 `event|null` 계약으로 확장되어 terminal failure 추적이 가능해졌다.
- `src/providers/event-normalizer.js`에도 같은 감지를 추가해 raw provider 이벤트가 normalizer 경로로 들어와도 `success` 결과가 실패로 바뀌게 했다.
- `src/progress-writer.js`는 `result/success` 이벤트가 권한 승인 대기 본문을 담으면 `progress.json`에 `status=failed`와 `last_failure`를 기록한다.
- `test/providers-failure.test.js`, `test/providers-normalizer.test.js`, `test/progress-writer.test.js`, `test/providers-claude.test.js`, `test/pipeline-runner.test.js`에 회귀 케이스를 추가했다.

## 결정 사항

- Claude 권한 승인 대기 응답은 `model_response` failure로 분류하고 code는 `claude_permission_request`로 둔다.
  실행 환경의 사용자 승인 UI를 headless run에서 처리할 수 없고, 모델이 실제 파일 변경을 완료하지 못한 상태를 성공으로 볼 수 없기 때문이다.
- `retryable=false`, `blocked=true`로 기록한다.
  같은 headless 설정으로 재시도해도 사용자 또는 provider 실행 정책 변경 전에는 해결되지 않는 blocked failure이기 때문이다.
- 감지는 raw provider, normalizer, progress writer의 세 경로에 둔다.
  provider 이벤트 경로가 바뀌거나 일부 계층이 우회되어도 `do-result.md`와 `progress.json`의 completed false-positive를 막기 위한 방어적 설계다.
- `last_error` 문자열의 하위 호환 의미는 유지하고, 구조화된 판단은 `last_failure`에 남긴다.

## 재발 방지 포인트

- provider `result(success)`는 항상 실제 작업 완료를 의미하지 않는다.
  본문에 권한 승인 대기, 사용자 승인 필요, 도구 실행 미완료 같은 문구가 있으면 success terminal로 기록하기 전에 failure classifier를 통과시킨다.
- headless provider는 interactive permission prompt를 처리할 수 없으므로 권한 대기 응답을 재시도 가능한 일반 model output으로 두지 않는다.
- provider raw adapter, event normalizer, progress writer 중 한 계층만 수정하면 event path drift로 같은 false-positive가 재발할 수 있다.
  권한 대기와 같은 blocked terminal 응답은 세 계층에서 같은 failure code로 수렴해야 한다.
- Do 단계 완료 판정은 provider exit code만 보지 않고 file contract 산출물과 `last_failure`를 함께 봐야 한다.
