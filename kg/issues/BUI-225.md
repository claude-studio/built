## 목표

Codex provider를 `workspace-write`로 실행할 때 허용되는 쓰기 범위와 금지해야 할 경계를 명확히 한다.
또한 `plan_synthesis`, `check`, `report` 같은 read-only phase에서 파일 변경 시도가 발생하면 성공처럼 지나가지 않고 sandbox failure로 감지되게 한다.

## 구현 내용

- `docs/contracts/provider-config.md`에 `workspace-write` 허용 범위를 추가했다.
  허용 범위는 feature 구현 파일, 테스트, 구현과 직접 연결된 문서, built runtime이 소유한 `.built/features/<feature-id>/` 결과물, runner/control plane 표준 writer 산출물로 제한했다.
- 같은 문서에 read-only phase 쓰기 금지 기준과 사용자 조치를 정리했다.
  `plan_synthesis`, `check`, `report`에서 Codex app-server가 `fileChange` notification을 보내면 built는 sandbox failure로 처리한다.
- `.git/`, credential/token 후보, local-only config, workspace 밖 경로, Multica agent runtime과 built provider runtime 경계가 섞이는 경로를 write-scope guard 후보로 문서화했다.
- `docs/ops/provider-routing-matrix.md`에 `do`/`iter`의 `workspace-write` 운영 범위와 guard 후보 참조를 보강했다.
- `src/providers/codex.js`는 read-only sandbox 실행 중 `fileChange` notification을 받으면 `codex_read_only_file_change` sandbox failure로 종료한다.
- `src/providers/failure.js`는 read-only phase 쓰기 시도와 `do`/`iter`의 read-only sandbox 충돌을 다른 사용자 조치 메시지로 분리한다.
- `test/providers-codex.test.js`에 `phase=check + sandbox=read-only` 중 `fileChange` notification 발생 시 실패하는 테스트를 추가했다.
- `test/providers-failure.test.js`는 새 sandbox failure 분류 메시지와 code를 검증한다.

## 결정 사항

- `workspace-write`는 `do`/`iter`에서 acceptance criteria 구현에 필요한 worktree 변경 권한으로 한정한다.
  provider가 파일 계약을 직접 소유하지 않고 표준 event를 emit하며, 파일 쓰기와 normalization은 runner/control plane이 담당한다는 기존 책임 경계를 유지하기 위해서다.
- read-only phase의 `fileChange` notification은 provider 품질 실패가 아니라 phase 목적과 sandbox 권한의 충돌로 본다.
  따라서 retry 대상이 아니며, 사용자 조치는 prompt/provider 설정을 수정하거나 실제 구현 변경을 `do`/`iter + workspace-write`로 옮기는 것이다.
- `.git`, credential, local-only config, workspace 밖 경로에 대한 보호는 이번 이슈에서 OS 수준 sandbox 신규 구현으로 확장하지 않고 guard 후보로 기록했다.
  이슈의 비범위가 모든 파일 시스템 접근 차단이었고, 현재 변경은 provider 실행 위험을 줄이는 문서/테스트/adapter failure 처리 보강이기 때문이다.

## 재발 방지 포인트

- read-only phase에서 Codex `fileChange` notification이 생기면 성공/경고로 흘리지 말고 sandbox failure로 종료한다.
- `do`/`iter` 구현 변경에 Codex를 쓰려면 `workspace-write`를 사용한다.
  `read-only`로 실행하면 실제 파일 변경이 반영되지 않아 성공처럼 보이는 잘못된 상태가 생길 수 있다.
- provider가 `.git/`, credential/token 후보, local-only config, workspace 밖 경로를 직접 수정하는 요구가 생기면 guard 후보를 실제 enforcement로 승격할지 별도 이슈와 검토를 거친다.
- provider sandbox failure 메시지를 추가할 때는 `src/providers/failure.js`, `docs/contracts/provider-config.md`, 관련 provider 테스트를 함께 갱신한다.
- `fileChange` notification 계약에 의존하는 감지는 OS 수준 sandbox가 아니다.
  workspace 밖 접근 차단이나 symlink escape 방지는 이번 완료 기준으로 충족된 것으로 간주하지 않는다.
