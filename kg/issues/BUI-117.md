## 목표

Codex를 built provider로 연결하기 위한 app-server 기반 provider adapter MVP를 추가한다.
CLI batch 호출이 아니라 Codex app-server의 thread/turn/progress 모델을 얇게 감싸고, built 표준 provider event와 runner/writer 경계에 맞춰 연결한다.

이번 범위는 실제 Codex smoke가 아니라 import/unit 테스트 가능한 control-plane adapter를 만드는 것이다.
plan_synthesis/do phase의 실제 Codex smoke와 usage/cost 정규화는 후속 이슈 범위로 남겼다.

## 구현 내용

- `src/providers/codex.js` 신규 추가:
  - `checkAvailability`로 `codex --version`과 `codex app-server --help`를 확인한다.
  - `checkLogin`으로 Codex 인증 상태를 확인한다.
  - `_AppServerJsonRpcClient` 최소 구현으로 app-server process를 spawn하고 JSON-RPC `initialize`, `thread/start`, `turn/start`, notification 수신을 처리한다.
  - Codex notification을 `phase_start`, `text_delta`, `tool_call`, `tool_result`, `phase_end`, `error` 표준 provider event로 변환한다.
  - provider는 `.built` 산출물 파일을 직접 쓰지 않고 `onEvent` 콜백과 result object만 반환한다.
- `src/pipeline-runner.js` 수정:
  - phase별 provider spec이 `codex`일 때 `runCodex` 경로를 사용한다.
  - Codex 이벤트는 `createStandardWriter` 경로를 통해 파일 계약으로 기록되도록 연결한다.
  - 현재 phase 값을 `runCodex`에 전달한다.
- `test/providers-codex.test.js` 신규 추가:
  - 실제 Codex 호출 없이 fake spawn/app-server fixture로 adapter 동작을 검증한다.
  - availability/login 실패 메시지, sandbox 변환, notification normalize, timeout, phase 주입, duration 보강, do/iter read-only 거부를 검증한다.

## 결정 사항

- `vendor/codex-plugin-cc`를 runtime dependency로 직접 묶지 않고 app-server JSON-RPC client 최소 구현을 채택했다.
  vendored package는 ESM과 Claude plugin 배치 전제를 갖고 있어 현재 CJS provider 모듈에서 안정적인 require 경계로 쓰기 어렵다.
- Codex sandbox 값은 명시적 변환 테이블로 유지한다.
  built 계약은 `read-only`/`workspace-write`이고 Codex app-server는 `readOnly`/`workspaceWrite`를 사용한다.
- 기본값은 `sandbox=read-only`, `approvalPolicy=never`, `timeout_ms=1800000`으로 둔다.
  단, `do`/`iter` phase에서 `read-only`는 파일 변경이 반영되지 않으므로 provider 진입 시 명확히 거부한다.
- timeout 경로에서는 terminal `error` emit과 settle을 먼저 보장하고, `turn/interrupt`는 best-effort cleanup으로 분리한다.
  app-server 무응답이 timeout 원인일 때 interrupt request를 await하면 provider promise가 끝나지 않을 수 있기 때문이다.
- 표준 이벤트의 `phase`는 `runCodex`의 `emit` wrapper에서 일괄 주입한다.
  `_notificationToEvents`는 phase를 모르는 순수 변환 함수로 유지하고, runner에 전달되는 이벤트 계약은 emit 시점에 보강한다.
- `phase_end.duration_ms`는 notification 변환 함수 내부가 아니라 `runCodex` emit 직전에 실제 경과 시간으로 채운다.

## 재발 방지 포인트

- timeout handler 안에서 app-server request를 await한 뒤 settle하면 무응답 app-server에서 provider promise가 hang될 수 있다.
  terminal event와 settle을 먼저 보장하고 interrupt/kill은 best-effort cleanup으로 둔다.
- `phase`는 writer 내부 상태만으로 보완하면 안 된다.
  외부 `onEvent` 소비자가 받는 표준 이벤트 자체에도 `phase`가 있어야 한다.
- `phase_end.duration_ms`는 성공 result meta뿐 아니라 표준 이벤트에도 채워야 한다.
  운영 디버깅과 provider-events 계약이 이벤트 payload를 기준으로 하기 때문이다.
- `do`/`iter`에서 Codex `read-only` sandbox를 허용하면 provider가 성공처럼 끝나도 파일 변경이 반영되지 않을 수 있다.
  parser 정책과 runtime provider 진입 검증을 함께 유지한다.
- vendored Codex plugin runtime을 직접 require하기 전에 module format과 plugin 배치 전제를 확인해야 한다.
  이번 MVP에서는 ESM/CJS 경계와 readline 호환성 문제 때문에 data 이벤트 기반 JSON-RPC 최소 구현이 더 낮은 위험이었다.
