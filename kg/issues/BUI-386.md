## 목표

`/built:run`이 execution worktree에서 성공한 뒤 사용자가 변경사항 위치와 root 적용/merge 다음 단계를 바로 알 수 있게 한다.
root working tree가 run 완료 시 자동 변경되지 않는 의도도 stdout, report, status, cleanup, doctor artifact에서 확인 가능해야 한다.

## 구현 내용

- `src/worktree-handoff.js`를 추가해 execution worktree 정보와 root 적용 상태를 평가하고, console/report handoff 문구를 생성하도록 했다.
- `scripts/run.js`가 run 완료 시 `state.execution_worktree.root_applied`, `root_apply_status`, `root_apply_summary`를 기록하고 stdout/report에 worktree branch, worktree path, result_dir, inspect/apply/merge/cleanup next step을 출력하도록 변경했다.
- `scripts/status.js`가 completed worktree run의 root 적용 상태와 handoff summary를 표시하도록 했다.
- `scripts/cleanup.js`가 cleanup 전 root 적용 상태, worktree branch/path, result_dir를 보여주도록 했다.
- `src/providers/doctor.js`가 completed worktree run 중 root 미적용 상태를 `worktree_handoff` warning으로 진단하도록 했다.
- `docs/ops/run-worktree-handoff.md`를 추가해 patch apply, branch merge, cleanup 절차를 문서화했다.
- run/status/cleanup/provider-doctor 단위 테스트를 추가하거나 보강했다.

## 결정 내용

- worktree-first run 완료 시 root working tree는 자동 적용하지 않는다.
- 사용자는 worktree diff를 확인한 뒤 patch apply 또는 worktree branch merge로 root 반영을 명시 수행한다.
- root 적용 상태는 `state.execution_worktree.root_applied`, `root_apply_status`, `root_apply_summary`로 기록하고, 후속 명령은 registry/state pointer를 우선해 같은 상태를 표시한다.
- `/built:apply` 별도 명령은 이번 범위에서 만들지 않고, 문서와 handoff 출력으로 다음 행동을 제공한다.

## 결정 이유

- 자동 apply는 root dirty 변경, 충돌, rollback 처리 없이 실행되면 사용자 작업을 예기치 않게 훼손할 수 있다.
- execution worktree-first 구조의 장점은 root 보호와 산출물 격리이므로, root 적용은 사용자 확인 이후의 명시 동작이어야 한다.
- stdout만으로는 사용자가 놓칠 수 있으므로 report, status, cleanup, provider-doctor가 같은 적용 상태를 반복 노출해야 한다.
- `state.execution_worktree`에 상태 필드를 남기면 cleanup이나 doctor처럼 나중에 실행되는 도구도 동일한 handoff 판단을 재사용할 수 있다.

## 완료 기준 충족 여부

| 기준 | 상태 |
|------|------|
| Run 성공 후 변경사항 위치를 바로 알 수 있음 | 완료, stdout/report/status에 worktree branch/path/result_dir가 노출됨 |
| 다음 merge/apply 절차를 바로 알 수 있음 | 완료, inspect, patch apply, branch merge, cleanup next step이 handoff에 포함됨 |
| root가 자동 변경되지 않는 의도 확인 가능 | 완료, report와 상태 요약에 root working tree 미변경 의도가 명시됨 |
| cleanup/doctor에서도 완료 worktree 적용 상태 확인 가능 | 완료, cleanup summary와 `worktree_handoff` warning으로 확인 가능 |

## 재발 방지 포인트

- worktree-first run 관련 UX를 바꿀 때는 root working tree가 자동 변경되지 않는 의도와 next step이 stdout/report/status/cleanup/doctor 중 최소 관련 경로에 남는지 확인한다.
- `status`와 `cleanup`은 root fallback보다 registry/state pointer를 우선해야 worktree result_dir와 root fallback을 혼동하지 않는다.
- root 적용 상태 필드를 바꿀 때는 `scripts/run.js`, `scripts/status.js`, `scripts/cleanup.js`, `src/providers/doctor.js`, `docs/ops/run-worktree-handoff.md`, 관련 테스트를 함께 갱신한다.
- 별도 `/built:apply`를 도입하려면 root dirty state, conflict recovery, rollback, branch merge policy를 먼저 결정해야 한다.
