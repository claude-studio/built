## 목표

`scripts/run.js`의 hook 실패 설명과 실제 pipeline 동작을 맞춘다.
Do/Check 전후 hook halt가 즉시 run 실패로 끝나지 않고 iter가 이해하는 `check-result.md` 복구 채널로 전달되는지 확정하고, 복구 가능한 hook 결과와 hard halt 결과를 구분한다.

## 구현 내용

- `scripts/run.js`에서 `before_do`, `after_do`, `before_check`의 `halt_on_fail: true` 실패를 `check-result.md` `needs_changes` 주입으로 정렬했다.
- `before_do` halt는 Do/Check를 건너뛰고, `after_do` halt는 Check를 건너뛰며, `before_check` halt는 Check를 건너뛰되 모두 iter 복구로 이어지게 했다.
- `before_report` halt는 구현 피드백 복구 채널이 아니므로 Report 전 hard halt로 유지했다.
- `src/hooks-runner.js`의 `injectFailuresIntoCheckResult()`가 신규 synthetic `check-result.md`에도 `feature`, `status`, `checked_at`, `provider`, `model`, `duration_ms`, `issues` frontmatter를 기록하게 정렬했다.
- `after_do` halt에서 `[hook-failure]`가 중복 주입되지 않도록 halted 분기 이후 warning 주입 경로를 타지 않게 했다.
- `test/run.test.js`에 `before_do`, `after_do`, `before_check`, `before_report` 회귀 테스트를 추가하고, `test/hooks-runner.test.js`에 synthetic `check-result.md` frontmatter 계약 검증을 보강했다.

## 결정 내용

- Do/Check 전후의 recoverable hook halt는 `check-result.md`를 단일 복구 채널로 사용한다.
- `check-result.md`는 정상 Check provider가 실행되지 않은 경우에도 최소 file contract frontmatter를 갖춘 synthetic artifact로 생성한다.
- `provider`와 `model`은 실제 Check provider가 실행되지 않았으면 `null`, `duration_ms`는 `0`으로 기록한다.
- Report 직전의 `before_report` halt는 구현 피드백을 iter에 전달하는 상황이 아니므로 run hard halt로 유지한다.

## 결정 이유

- iter loop는 `check-result.md`의 `status: needs_changes`와 `issues`를 기준으로 다음 Do 수정을 판단한다.
  Do/Check 전후 hook 정책 위반도 이 채널로 들어와야 자동 수정 흐름이 끊기지 않는다.
- hook 실패를 provider terminal event 실패로 소급하면 provider event SSOT와 runner 산출물 책임 경계가 뒤섞인다.
- synthetic artifact라도 frontmatter 계약이 비어 있으면 file contract 소비자가 정상 Check 결과와 복구용 결과를 일관되게 읽을 수 없다.
- Report 전 halt는 이미 구현/검증 루프를 위한 피드백이 아니라 최종 보고 단계 진입 차단이므로, iter 복구 요청으로 가장하면 의미가 흐려진다.
