## 목표

provider, check, iter, report 테스트에서 반복되는 feature spec fixture를 일관되게 만들 수 있도록 테스트 전용 generator를 추가한다.
acceptance criteria, excludes, build_files, provider config 변형을 쉽게 만들고, 생성된 fixture frontmatter가 실제 file contract와 어긋나지 않게 검증 가능한 형태로 둔다.

## 구현 내용

- `test/fixtures/feature-spec-generator.js`를 추가해 feature spec markdown, do-result/check-result frontmatter, provider config, Codex do config를 생성하게 했다.
- `makeFeatureSpecProject()`가 임시 프로젝트 루트와 `.built/features/<feature>.md`를 만들고, `writeRunRequest()`가 `.built/runtime/runs/<feature>/run-request.json`을 작성하게 했다.
- `assertFeatureSpecFrontmatter()`, `assertDoResultFrontmatter()`, `assertCheckResultFrontmatter()`를 제공해 테스트 fixture가 계약 필드를 누락하거나 잘못된 status를 쓰면 빠르게 실패하게 했다.
- `test/feature-spec-generator.test.js`를 추가해 기본값, override, 임시 프로젝트 생성, run-request 작성, frontmatter 검증 실패 사례를 34개 단위 테스트로 고정했다.

## 결정 사항

- generator는 외부 npm 패키지 없이 Node.js 내장 모듈만 사용한다.
  테스트 fixture 생성이 provider 실행이나 YAML parser 의존성에 묶이면 provider 전환 회귀를 확인하기 전에 fixture 계층 자체가 흔들릴 수 있기 때문이다.
- feature spec frontmatter는 `BUILT-DESIGN.md` 7장 기준 필수 필드인 `feature`, `version`, `created_at`, `confirmed_by_user`, `status`, `primary_user_action`, `excludes`, `build_files`를 기본 생성한다.
  provider/check/iter/report 테스트가 같은 입력 계약을 공유해야 phase별 테스트 차이만 드러난다.
- do-result/check-result frontmatter 검증은 최소 계약과 status enum을 우선 고정한다.
  비용/usage 같은 optional telemetry는 fixture 필수 조건으로 승격하지 않고, 산출물 계약에서 꼭 필요한 필드만 기본값으로 둔다.
- `buildProviderConfig()`는 기본 provider config를 얕은 override로 확장한다.
  테스트에서 phase별 provider 변형을 짧게 만들되, 중첩 provider spec의 깊은 병합 정책을 generator의 숨은 동작으로 만들지 않기 위해서다.

## 재발 방지 포인트

- provider 관련 테스트가 직접 feature spec frontmatter를 손으로 만들기 시작하면 계약 drift가 다시 생긴다.
  새 provider/check/iter/report 테스트는 가능한 한 `test/fixtures/feature-spec-generator.js`의 generator와 assert helper를 공유한다.
- optional telemetry 필드를 fixture 필수 조건처럼 만들지 않는다.
  do-result 최소 계약과 usage/cost optional 정책을 혼동하면 telemetry 부재가 테스트 실패로 번질 수 있다.
- `buildProviderConfig()`의 override는 얕은 병합이다.
  중첩 provider spec 일부만 바꾸려는 테스트는 명시적으로 전체 nested object를 넘겨야 한다.
- 기존 테스트 중복 fixture 정리는 아직 후속 작업으로 남아 있다.
  이번 generator가 추가되었더라도 기존 fixture를 무리하게 일괄 교체하지 말고 provider/check/iter/report 변경 시점에 점진적으로 흡수한다.
