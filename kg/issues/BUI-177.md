## 목표

Claude Code plugin 형태로 설치되는 built package에서 metadata, marketplace 경로, skill 문서의 script 참조, provider 관련 문서와 helper가 누락 없이 연결되는지 검증한다.
로컬 `--plugin-dir` 방식과 repository-root 실행 방식 사이의 경로 차이로 사용자 설치 후 skill 또는 script 호출이 깨지는 문제를 조기에 잡는 것이 목표다.

## 구현 내용

- `test/plugin-packaging.test.js`를 추가해 외부 npm 패키지 없이 Node.js 기본 `assert`, `fs`, `path`만으로 packaging smoke를 실행하게 했다.
- 루트 `.claude-plugin/plugin.json`과 `plugins/built/.claude-plugin/plugin.json`의 필수 metadata를 검증한다.
- `plugins/built/scripts`, `plugins/built/skills`, `plugins/built/src`가 루트 디렉토리를 가리키는 심볼릭 링크인지 확인한다.
- `skills/*/SKILL.md`에서 참조하는 `../../scripts/<name>.js`와 `node scripts/<name>.js` 경로가 실제 `scripts/` 파일과 일치하는지 검증한다.
- provider 배포에 필요한 skill, script, 문서가 package에서 빠지지 않았는지 확인한다.
  대상은 `doctor`, `run`, `run-opus`, `run-sonnet` skill, provider doctor/preset/compare/smoke script, provider config/events/file contracts/setup/smoke 문서다.
- `plugins/built` plugin metadata의 skills 경로가 실제 skills 디렉토리와 일치하는지 확인한다.
- `--plugin-dir` 실행 관점의 `plugins/built/scripts`, `plugins/built/skills` 파일 목록이 repository-root의 `scripts`, `skills`와 동등한지 확인한다.
- `README.md`에 적힌 `npm run`/`npm test` 명령이 `package.json` scripts에 존재하는지 확인한다.

## 결정 내용

- plugin packaging smoke는 실제 marketplace publish 자동화가 아니라 repository 안의 packaging 계약 검증으로 둔다.
- skill 문서의 script 참조 허용 패턴은 plugin 설치 경로용 `../../scripts/<name>.js`와 repository-root 개발용 `node scripts/<name>.js` 두 축으로 검증한다.
- provider 기능 배포 체크리스트는 doctor, preset, compare, real smoke, Codex smoke script와 provider 5개 기준 문서 존재 여부를 포함한다.
- `plugins/built`는 scripts, skills, src를 루트 디렉토리로 연결하는 심볼릭 링크 구조를 유지해야 한다.

## 결정 이유

- built는 사용자에게 Claude Code plugin으로 전달되므로 repository-root에서만 통과하는 테스트로는 설치 후 경로 오류를 잡기 어렵다.
- skill 문서가 script 경로를 직접 안내하므로 문서 참조와 실제 파일 구조를 함께 검증해야 사용자-facing 명령 drift를 줄일 수 있다.
- provider 기능은 여러 skill, helper script, 계약 문서에 걸쳐 노출되므로 단일 구현 테스트만으로는 package 누락을 발견하기 어렵다.
- marketplace publish 자동화 없이도 metadata, symlink, 문서, script 존재 계약을 고정하면 release 전 smoke 기준으로 재사용할 수 있다.

## 재발 방지 포인트

- 새 skill이 `scripts/` helper를 참조하면 `../../scripts/<name>.js` 또는 `node scripts/<name>.js` 패턴이 실제 파일과 일치해야 한다.
- provider 관련 새 helper나 smoke script가 추가되면 packaging smoke의 provider 필수 목록도 함께 검토한다.
- `plugins/built`의 symlink 구조를 바꿀 때는 `--plugin-dir` 방식과 repository-root 방식의 파일 목록 동등성을 다시 확인한다.
- README에 새 `npm run` 명령을 추가하면 `package.json` scripts 정의를 함께 추가한다.
- packaging smoke는 external publish를 대신하지 않으며, publish 자동화가 생기면 이 smoke를 release gate의 사전 검증으로 연결한다.
