## 목표

`scripts/report.js`를 직접 실행할 때도 `/built:run` 경로와 같은 검증 순서를 강제한다.
approved `check-result.md` 없이 `report.md`가 조용히 생성되는 경로를 막고, 검증 미완료 보고서가 필요한 예외 상황은 명시 opt-in과 artifact evidence로 남긴다.

## 구현 내용

- `scripts/report.js`에 `check-result.md` frontmatter `status: approved` gate를 추가했다.
- `check-result.md`가 없거나 `status: needs_changes`이면 기본적으로 provider 실행 전 실패하도록 했다.
- `--allow-unchecked` 플래그를 추가해 검증 미완료 보고서 생성을 명시 opt-in으로 제한했다.
- 예외 실행 시 `report.md` frontmatter에 `check_status`, `unchecked: true`, `unchecked_reason`을 기록하도록 했다.
- `test/report.test.js`에 missing check-result, needs_changes, approved, allow-unchecked evidence 회귀 테스트를 추가했다.
- `skills/report/SKILL.md`와 `docs/contracts/file-contracts.md`에 direct report gate와 예외 artifact 계약을 문서화했다.

## 결정 내용

- direct `/built:report`의 기본 선행조건은 `check-result.md.status == approved`다.
- missing `check-result.md` 또는 `needs_changes` 상태에서는 기본적으로 `report.md`를 생성하지 않는다.
- 검증 미완료 보고서가 필요한 경우 `--allow-unchecked`를 사용해야 하며, 결과 artifact가 unchecked 상태와 사유를 자체적으로 설명해야 한다.

## 결정 이유

- `report.md`는 Plan/Do/Check/Report artifact 흐름의 완료 보고서라서 Check 승인 여부와 분리되면 완료 의미가 흐려진다.
- `/built:run` orchestrator 경로와 direct script 경로의 gate가 다르면 같은 feature에 대해 서로 다른 safety contract가 생긴다.
- 예외 실행을 완전히 금지하면 운영상 필요한 진단 보고서를 만들 수 없으므로, opt-in과 frontmatter evidence를 함께 요구하는 방식이 추적성과 유연성을 균형 있게 보장한다.

## 재발 방지 포인트

- phase result markdown을 직접 생성하는 script는 orchestrator 경로와 같은 선행조건을 자체적으로 검사해야 한다.
- `report.md` 생성 조건을 바꿀 때는 `check-result.md` missing, `needs_changes`, `approved`, `--allow-unchecked` 케이스를 분리해 테스트한다.
- 검증 미완료 예외는 silent bypass가 아니라 CLI flag와 artifact frontmatter evidence를 함께 요구한다.
- provider는 built 결과 파일을 직접 쓰지 않는다는 계약을 유지하고, gate와 frontmatter 정규화는 runner/helper 계층에서 처리한다.
