## 목표

사용자가 `/built:status`에서 실패 상태를 봤을 때 어떤 provider와 phase에서 실패했는지,
어떤 model을 썼는지, 그리고 다음에 무엇을 해야 하는지 바로 판단할 수 있게 한다.
usage/cost 정보가 없는 provider 또는 smoke artifact는 오류처럼 보이지 않고 `미제공`으로 표시해야 한다.

## 구현 내용

- `scripts/status.js`가 실패 상태에서 `provider`, `phase`, `model`, `failure.kind`,
  `failure.code`, `action(next_action)`, `retryable`, `blocked`를 함께 출력하게 했다.
- `last_failure.action`은 사용자-facing 다음 조치 문장으로 노출하고, 출력 라벨은
  `action(next_action)`으로 정리했다.
- `last_failure.code === "claude_permission_request"`이면 기존 generic action을 숨기고
  "아래 remediation 중 하나를 선택하세요."를 표시한 뒤 `/built:run-codex <feature>` 재실행 또는
  `.claude/settings.json` allow rule 추가 선택지를 보여준다.
- `last_error`가 객체일 때 `debug_detail`과 `raw_provider`는 status 출력에서 제외하고,
  긴 값은 한 줄로 축약한다.
- KST 절대 시각과 상대 시각을 함께 표시하도록 `heartbeat`, `started`, `updated`, list updated
  출력 경로를 정리했다.
- progress의 `cost_usd`, `input_tokens`, `output_tokens`가 없으면 `미제공`으로 표시한다.
- `test/status.test.js`에 failure summary, permission remediation, usage/cost 미제공, KST 시간,
  debug detail 비노출 회귀 테스트를 추가했다.

## 결정 사항

- status는 provider failure의 디버그 전문을 보여주는 화면이 아니라 사용자 조치 화면으로 둔다.
  따라서 `debug_detail`과 raw provider payload는 logs 계층에서 확인하게 하고 status에는 요약 필드만 둔다.
- `retryable`과 `blocked`는 자동 실행 지시가 아니라 실패 판단 신호로 표시한다.
  사용자가 실제로 실행할 문장은 `action(next_action)`에 둔다.
- usage/cost telemetry가 없는 상태는 실패가 아니다. provider가 값을 제공하지 않았다는 의미로
  `미제공`을 표시한다.
- Claude permission request는 일반 failure action보다 전용 remediation 선택지가 중요하다.
  기존 generic action은 숨기고, 사용자가 선택 가능한 재실행/allow rule 안내를 노출한다.

## 재발 방지 포인트

- status 출력이 실패 요약을 바꿀 때 `failure.kind/code/action(next_action)/retryable/blocked` 중
  일부만 보여주면 사용자가 조치 문장과 orchestration 신호를 함께 판단하기 어렵다.
- `claude_permission_request`는 기존 generic action을 그대로 보여주면 실제 선택지인
  Codex 재실행 또는 allow rule 안내가 묻힐 수 있다.
- `debug_detail`, `raw_provider`, 긴 raw error는 status에 직접 노출하지 않는다.
  상태 화면은 요약과 다음 조치 중심으로 유지하고, 원문 디버그는 logs 계층에 둔다.
- cost/usage 값 부재는 telemetry 미제공으로 취급한다. 실패나 0 비용으로 오해되게 출력하지 않는다.
