## 목표

provider 전환의 첫 구현 단계로, 기존 Claude 직접 호출 동작을 깨지 않고
`src/providers/claude.js` 형태로 추출하기 전에 핵심 산출물 계약을 테스트로 고정한다.

## 구현 내용

- `src/providers/claude.js` 신규 추가: Claude CLI 호출 로직을 `runClaude({ prompt, model, onEvent, jsonSchema })` 인터페이스로 추출.
- `src/pipeline-runner.js` 수정: 직접 `spawn` 호출 대신 `runClaude`와 `createWriter` 조합을 사용하도록 단순화.
- `test/file-contracts.test.js` 신규 추가: `state.json`, `progress.json`, `do-result.md` 핵심 산출물 contract/snapshot 테스트 27개 추가.
- `test/providers-claude.test.js` 신규 추가: Claude provider 단위 테스트 22개 추가.
- `docs/contracts/file-contracts.md`, `docs/contracts/provider-events.md`, `docs/contracts/provider-config.md`와 provider 전환 roadmap 문서를 추가.

외부 CLI 호출, 파일 쓰기 위치, 진행 상태 기록은 기존 동작을 유지했다.
`_parseTimeout`은 기존 테스트와 하위 호환을 위해 `pipeline-runner.js`에서 재익스포트한다.

## 결정 사항

- provider는 파일을 직접 쓰지 않고 `onEvent` 콜백으로 이벤트만 전달한다.
  파일 normalize/write 책임은 runner와 writer 계층에 둔다.
- `pipeline-runner.js`는 provider API(`runClaude`)만 사용하고 `childProcess.spawn`을 직접 참조하지 않는다.
- `json-schema` 모드도 provider에 위임하되, 기존 `runtimeRoot`/`featureId` 미사용 동작은 유지한다.
- 테스트 호환성을 위해 `_parseTimeout`을 runner에서 재익스포트한다.

## 발생한 이슈

- Review 라운드 1에서 Pass.
- Reviewer 확인 사항:
  - `writer.handleLine`에서 `writer.handleEvent`로의 전환이 올바르게 적용됨.
  - provider가 JSON 파싱 후 이벤트 객체를 `onEvent`로 전달하고, `progress-writer.js`의 기존 `handleEvent` API와 호환됨.
  - `writer.close()`가 성공/실패 양 경로에서 호출되어 리소스 누수 없음.
  - 기존 `childProcess.spawn` mock이 모듈 캐시 공유를 통해 `providers/claude.js`에도 적용되어 기존 테스트 방식 유지 가능.

## 재발 방지 포인트

- provider는 이벤트 생성과 CLI 실행만 담당한다. `state.json`, `progress.json`, `do-result.md` 쓰기 책임을 provider로 되돌리지 않는다.
