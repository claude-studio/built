## 목표

interactive discovery는 host가 담당하고, 질문/답변/컨텍스트를 정리한 뒤 provider가 구현 계획을 구조화하는 `plan_synthesis` phase를 코드 경로에 연결한다.
Claude와 Codex가 같은 입력 계약과 산출물 계약을 사용해야 하며, 실제 Codex smoke는 기본 CI가 아니라 명시적 opt-in으로만 실행되어야 한다.

## 구현 내용

- `src/plan-synthesis.js`를 추가해 `plan_synthesis` 입력 payload 생성, provider prompt 생성, provider output 정규화, canonical 산출물 기록 helper를 제공했다.
- 입력 payload는 `feature_id`, `feature_spec_path`, `feature_spec`, `questions`, `answers`, `repo_context`, `prior_art`, `acceptance_criteria`, `constraints`를 포함한다.
- 산출물은 `.built/features/<feature>/plan-synthesis.json`과 `.built/features/<feature>/plan-synthesis.md`에 기록한다.
  JSON 필수 구조는 `feature_id`, `phase`, `provider`, `model`, `created_at`, `output`이며, `output`은 `summary`, `steps`, `acceptance_criteria`, `risks`, `out_of_scope`를 포함한다.
- `scripts/plan-synthesis.js`를 추가해 provider 실행 결과를 정규화하고 runner 계층에서 파일을 기록하게 했다.
- `scripts/run.js`는 `run-request.json`의 `plan_synthesis: true` 또는 `providers.plan_synthesis` 설정이 있을 때만 `do` 앞에서 `plan_synthesis`를 실행한다.
- `scripts/do.js`는 `.built/features/<feature>/plan-synthesis.json`이 있을 때 해당 output을 implementation plan으로 prompt에 주입한다.
- `scripts/smoke-codex-plan-synthesis.js`를 추가해 실제 Codex smoke를 `BUILT_CODEX_PLAN_SYNTHESIS_SMOKE=1` 환경 변수로 opt-in 분리했다.
- `src/providers/codex.js`는 `phase_end` 이벤트의 `result`에 최종 text를 연결해 `standard-writer`가 phase result를 동일 계약으로 기록할 수 있게 했다.
- `test/plan-synthesis.test.js`는 payload, prompt, output normalization, canonical 파일 기록, fake Codex 표준 이벤트의 `plan_synthesis` progress/result 계약을 검증한다.

## 결정 사항

- `plan_synthesis`는 provider가 파일을 수정하지 않는 read-only phase로 유지한다.
  provider는 계획 JSON/text만 반환하고 `.built` 파일 기록은 `src/plan-synthesis.js` helper와 runner script가 담당한다.
- activation은 opt-in으로 제한한다.
  legacy `run-request.json`은 기존 `do -> check -> iter -> report` 흐름을 유지하고, `plan_synthesis: true` 또는 `providers.plan_synthesis`가 있을 때만 0번째 phase가 실행된다.
- 실제 Codex smoke는 기본 테스트에 포함하지 않는다.
  Codex login, app-server, 네트워크/로컬 환경 의존성이 있는 경로이므로 `BUILT_CODEX_PLAN_SYNTHESIS_SMOKE=1`로 명시 실행할 때만 수행한다.
- `do` phase는 provider별 plan을 다시 만들지 않고 canonical `plan-synthesis.json`의 `output`만 읽는다.
  이렇게 하면 plan provider와 implementation provider가 달라도 같은 구현 계획을 공유한다.

## 재발 방지 포인트

- `plan_synthesis`를 기본 pipeline에 무조건 삽입하면 기존 사용자 flow와 CI 시간이 변할 수 있다.
  activation 조건은 `plan_synthesis: true` 또는 `providers.plan_synthesis`로 유지한다.
- 실제 Codex smoke를 기본 테스트에 넣으면 인증/로컬 app-server 상태에 따라 CI가 흔들릴 수 있다.
  기본 경로는 fake provider/file contract test로 보호하고, 실제 smoke는 환경 변수 opt-in으로 분리한다.
- provider가 `.built/features/<feature>/plan-synthesis.*`를 직접 쓰면 Claude/Codex 간 산출물 drift가 생긴다.
  provider는 result/event만 반환하고 runner/helper가 canonical 파일을 쓴다.
- `phase_end.result`가 비어 있으면 `standard-writer`가 provider 최종 text를 phase result로 남기지 못한다.
  Codex adapter 수정 시 terminal event의 result 연결을 회귀 테스트로 확인한다.
- `do` phase가 `plan-synthesis.json` 전체 메타를 prompt에 넣으면 provider/model/timestamp 같은 실행 메타가 구현 계획으로 섞일 수 있다.
  `readPlanSynthesisOutput`처럼 `output`만 읽어 사용한다.
