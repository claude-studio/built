## 목표

Claude/Codex 이후 새 provider를 붙일 때 필요한 adapter 골격과 계약 테스트를 표준화한다.
새 provider 추가자가 `runWorker` 형태, 표준 이벤트 emit, failure taxonomy, sandbox 지원, 파일 직접 쓰기 금지 규칙을 같은 기준으로 확인할 수 있게 만드는 것이 목표였다.

## 구현 내용

- `src/providers/scaffold-template.js`를 추가해 신규 provider adapter의 출발점을 제공했다.
  `runScaffold({ prompt, model, onEvent, sandbox, timeout_ms, signal })` 형태, `SCAFFOLD_TODO` 마커, `phase_start`/`phase_end`/`error` 이벤트 helper, `createFailure`/`failureToEventFields` 사용 예시를 포함한다.
- `docs/providers/new-provider-checklist.md`를 추가해 `VALID_PROVIDERS` 등록, provider 단위 테스트, 표준 이벤트 순서, failure kind 매핑, sandbox 처리, 파일 직접 쓰기 금지를 체크리스트로 정리했다.
- `test/providers-compliance.test.js`를 추가해 실제 제3 provider 없이도 fake adapter로 contract 위반을 검출하게 했다.
  파일 직접 쓰기 guard, 이벤트 순서 위반, terminal 이벤트 누락, `phase_end`와 `error` 동시 emit, AbortSignal 지원을 검증한다.

## 결정 사항

- 새 provider adapter는 `scaffold-template.js`를 복사해 시작하고, provider별 구현은 `SCAFFOLD_TODO`를 제거하는 방식으로 완성한다.
  기존 Claude/Codex adapter를 그대로 복제하면 provider별 raw runtime 차이와 built 표준 계약이 섞일 수 있기 때문이다.
- provider는 파일을 직접 쓰지 않고 `onEvent`로 표준 이벤트만 emit한다.
  파일 기록 책임은 `pipeline-runner.js`, `progress-writer.js`, standard writer 계층에 남겨 provider 전환 중 file contract drift를 막는다.
- compliance 테스트는 실제 provider CLI나 네트워크 없이 fake adapter로 유지한다.
  신규 provider가 추가되기 전에도 CI에서 계약 위반 패턴을 빠르게 고정할 수 있기 때문이다.

## 재발 방지 포인트

- 새 provider adapter가 `fs.writeFile`, `fs.appendFile`, write flag `fs.open`을 직접 사용하면 runner/writer 책임 경계가 깨진다.
  provider는 표준 이벤트만 emit하고 파일 기록은 runner 계층에 맡긴다.
- `phase_end`와 `error`는 둘 중 하나만 terminal 이벤트로 emit한다.
  둘 다 emit하거나 terminal 이후 이벤트를 보내면 progress/result 최종 상태가 뒤집힐 수 있다.
- `unknown` failure를 사용할 때는 `sanitizeDebugDetail()`을 거친 debug detail을 남기되 token, private environment value, raw path 후보를 사용자-facing 메시지에 넣지 않는다.
- checklist는 새 provider의 필수 등록 위치를 안내하지만, 실제 provider가 추가될 때는 `VALID_PROVIDERS`, runner 분기, normalizer, provider config contract를 별도 PR에서 함께 검증해야 한다.
