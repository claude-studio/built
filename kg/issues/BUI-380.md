## 목표

`halt_on_fail: false`인 `after_do`와 `before_check` hook warning이 Check phase의 `check-result.md` 재작성 이후에도 최종 check/report evidence에 남도록 한다.

## 구현 내용

- `src/hooks-runner.js`에 non-halt hook warning을 pending warning artifact로 저장하고 병합 대상으로 읽는 흐름을 추가했다.
- `scripts/check.js`가 새 `check-result.md`를 작성한 뒤 pending hook warning을 `issues` frontmatter와 본문 `Hook 경고 내역`에 병합하고 pending artifact를 제거하도록 했다.
- `scripts/run.js`에도 fallback 병합 경로를 두어 Check writer 이후에도 warning evidence가 누락되지 않게 했다.
- `test/hooks-runner.test.js`에 pending warning artifact 생성/병합 회귀 테스트를 추가했다.
- `test/run.test.js`에 `after_do`와 `before_check` non-halt warning이 최종 `check-result.md`와 `report.md` evidence에 남는 테스트를 추가했다.
- `docs/contracts/provider-events.md`에 `halt_on_fail: true/false` 차이와 non-halt Do/Check 전 hook warning 보존 계약을 명시했다.

## 결정 내용

- Do/Check 전 non-halt warning은 Check가 `check-result.md`를 다시 쓰기 전까지 pending artifact로 보관한다.
- Check writer는 pending warning을 최종 `check-result.md`의 `issues`와 본문 `Hook 경고 내역`에 병합한다.
- `before_do`, `after_do`, `before_check`의 `halt_on_fail: false` warning은 pipeline을 계속 진행시키며 `status`를 `needs_changes`로 강제하지 않는다.
- Report phase는 별도 hook feedback 파일이 아니라 최종 `check-result.md`를 통해 warning evidence를 소비한다.

## 결정 이유

- 기존 경로는 Check 전 hook warning을 `check-result.md`에 먼저 주입해도 Check provider가 새 파일을 작성하면 evidence가 덮어써질 수 있었다.
- `check-result.md`는 Iter와 Report가 공유하는 복구/판단 채널이므로 최종 파일에 warning이 남아야 운영자가 hook 경고를 artifact만 보고 추적할 수 있다.
- non-halt warning은 실패 판정이 아니라 감사 evidence이므로 `needs_changes`를 강제하면 `halt_on_fail: false` 계약과 맞지 않는다.
- provider event writer나 provider terminal state를 hook warning 보존 채널로 쓰면 provider 실행 결과와 사용자 자동화 hook 책임 경계가 섞인다.

## 재발 방지 포인트

- Check 전에 발생한 hook warning은 즉시 최종 `check-result.md`에만 쓰면 안 된다. Check writer가 파일을 다시 만들 수 있으므로 pending warning 병합 경로를 함께 유지한다.
- `halt_on_fail: false` warning 보존 테스트는 `check-result.md`의 `issues`와 본문, Report evidence를 함께 검증한다.
- non-halt warning은 감사/추적 evidence이며 실패 판정 강제 수단이 아니다. `status: needs_changes`는 recoverable halt 또는 실제 Check 판단 경로에서만 사용한다.
- hook warning 보존 정책을 바꿀 때는 `docs/contracts/provider-events.md`의 Hook 실패 영향 계약과 `kg/workflows/provider-aware-hook-validation.md`를 함께 갱신한다.
