## 목표

`.built/config.json`의 `default_run_profile.providers`를 사람이 읽고 수정하는 기본 실행 구성으로 고정한다.
config에는 ProviderSpec 객체를 저장하지 않고 `do`/`check`/`iter`/`report` phase별 provider name 문자열만 저장해야 한다.
feature 실행 시점의 `run-request.json`은 이 문자열 기본값을 normalized ProviderSpec snapshot으로 변환할 수 있어야 한다.

## 구현 내용

- `scripts/init.js`가 새 `.built/config.json`에 all-Claude 문자열 기반 `default_run_profile.providers`를 생성하도록 했다.
- `/built:validate` config 검증에서 `default_run_profile` known key를 허용하고, phase 누락, 알 수 없는 provider name, ProviderSpec 객체 입력을 명확한 오류로 처리했다.
- `src/providers/config.js`에 문자열 default profile을 run-request ProviderSpec으로 정규화하는 helper를 추가했다.
- Codex 문자열 profile을 snapshot으로 만들 때 `do`/`iter`는 `workspace-write`, `check`/`report`는 `read-only` sandbox를 자동 적용하도록 했다.
- README, provider config 계약 문서, provider setup guide, `skills/init`, `skills/run`에 default profile과 run-request snapshot의 역할 차이를 반영했다.
- init, validate, provider config, provider preset 테스트에 문자열 profile과 object ProviderSpec 실패 케이스를 추가했다.

## 결정 내용

- `default_run_profile.providers`는 ProviderSpec이 아니라 provider name 문자열 맵이다.
- `.built/config.json`에는 top-level `providers`를 두지 않고, `default_run_profile.providers`에도 `sandbox`, `model`, `timeout_ms` 같은 ProviderSpec detail을 저장하지 않는다.
- `run-request.json`은 feature별 실행 snapshot이며, 기본값 저장소로 쓰지 않는다.
- Codex sandbox는 config 저장값이 아니라 snapshot 생성 시 phase capability 정책에서 파생한다.

## 결정 이유

- config default profile은 사람이 직접 읽고 수정하는 기본값이므로 단순한 provider name 문자열이어야 drift가 적다.
- ProviderSpec 객체를 config에 허용하면 config schema와 run-request snapshot schema가 섞이고, sandbox 정책의 책임 위치가 불분명해진다.
- Codex sandbox를 capability 정책에서 파생하면 쓰기 phase와 읽기 phase의 권한 차이를 테스트 가능한 helper 계약으로 유지할 수 있다.
- 기존 run-request provider preset helper와 BUI-315의 `codex-run` 계약은 feature별 snapshot 생성 경계에 남아야 한다.

## 재발 방지 포인트

- config default profile에 ProviderSpec 객체를 허용하지 않는다.
- Codex sandbox default를 config에 저장하지 않고, snapshot 생성 helper와 capability 정책 테스트로 고정한다.
- provider preset helper는 run-request snapshot 생성 경계에 남기고, `.built/config.json`을 feature별 실행 결과나 runtime state 저장소로 쓰지 않는다.
- provider 문서에서 “config에 provider 설정을 쓰지 않는다”라고 표현할 때는 top-level `providers`와 ProviderSpec detail 금지인지, 문자열 default profile 허용인지 구분한다.
