## 목표

Queue Tick 또는 backlog drain이 built 프로젝트 기준 ready backlog를 0건으로 판단할 때 `project_id` 누락 가능성을 함께 진단하게 한다. 새 Queue Tick과 built backlog 생성 시 프로젝트 이름 추정이 아니라 실제 built `project_id`를 사용하도록 운영 지침을 명확히 한다.

## 구현 내용

- `docs/ops/queue-project-id-diagnostics.md`를 추가해 built `project_id`, backlog 점검 명령, 누락 의심 이슈 필터, 페이지네이션 확인, 한글/KST 코멘트 템플릿을 문서화했다.
- `kg/agents/coordinator.md`에 Queue Tick 종료 전 built project_id 기준 ready backlog 수와 project_id 누락 의심 건수를 코멘트에 남기는 절차를 보강했다.
- `kg/agents/operator.md`와 `kg/agents/finisher.md`에 Queue Tick 또는 Queue Recovery Tick 생성 시 실제 built `project_id`를 직접 지정하고 생성 후 `project_id`를 확인하는 규칙을 추가했다.
- `kg/workflows/feature-development-loop.md`에 Queue Tick 생성/확인 기준을 추가했다.

## 결정 내용

- built 프로젝트의 기준 `project_id`는 `068c9ad8-8efe-4692-9bf7-3521ddc06588`로 명시한다.
- ready backlog 0건 종료는 단순 종료가 아니라 `project_id` 누락 의심 backlog 점검을 포함한 evidence를 남긴 뒤 판단한다.
- `project_id`가 비어 있고 `BUI-` 식별자를 가진 backlog는 built 관련 누락 의심 후보로 본다.
- 새 Queue Tick과 built backlog는 프로젝트 이름이 아니라 실제 `project_id`를 사용해 생성한다.
- Builder, Reviewer, Recorder, Finisher는 임의로 기존 이슈의 project 보정을 수행하지 않고, 누락 의심 건이 있으면 Coordinator 판단으로 넘긴다.

## 결정 이유

- BUI-172 운영에서 project_id 없이 생성된 backlog가 built 프로젝트 기준 조회에서 제외되어 Queue Tick이 ready backlog를 0건으로 오판할 수 있음이 확인됐다.
- 프로젝트 이름 기반 생성은 서버 매칭 정책이나 CLI 동작 변화에 취약하므로 실제 `project_id`를 지정하는 쪽이 재현성과 감사 가능성이 높다.
- Queue Tick은 운영 라우팅의 마지막 단계이므로, ready 0건 판단 근거를 한글/KST 코멘트로 남겨 다음 Coordinator가 같은 원인을 반복 조사하지 않게 해야 한다.
- Multica 서버의 프로젝트 매칭 로직 변경은 이슈 비범위였으므로 운영 지침과 진단 evidence를 보강하는 방식이 범위에 맞다.
