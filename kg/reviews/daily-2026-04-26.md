# Daily Alignment Review

## 오늘의 정렬 판정

오늘 작업은 북극성 목표에 대체로 정렬되어 있다.
핵심 진전은 provider-agnostic control plane을 실제 사용자 배포와 실패 판정까지 확장한 점이다.
다만 운영 큐가 자동화 진행을 유지하기 위해 Queue Tick과 heartbeat에 강하게 의존하고 있어, 제어면 자체의 안정성보다 운영 보정 루프가 앞서는 드리프트가 남아 있다.

## 목표와 직접 연결된 진전

- [[issues/BUI-268]]은 headless Claude가 권한 승인 대기 응답을 성공 완료로 기록하던 false-positive를 `claude_permission_request` failure로 정규화했다.
  provider exit code보다 terminal event와 산출물 계약을 우선하는 방향이 북극성의 "신뢰 가능한 완료 상태" 기준에 맞다.
- [[issues/BUI-279]]는 Claude `result.is_error=true` 또는 `subtype=error`가 exit 0과 함께 오더라도 `success:false`로 승격되게 했다.
  `progress.json`, provider runner 반환값, `docs/contracts/provider-events.md`의 실패 의미가 같은 방향으로 수렴했다.
- [[issues/BUI-169]]는 provider preset helper와 skills 문서를 정리해 사용자가 수동 JSON 편집 없이 대표 provider 조합을 적용할 수 있게 했다.
  `claude-default`, `codex-do`, `codex-plan`, `codex-all` preset은 Claude 기본 경로를 보존하면서 Codex opt-in을 명시적으로 제공한다.
- [[issues/BUI-174]]는 real provider smoke 결과를 `.built/runtime/smoke/<id>/summary.json` artifact로 남기고 redaction, taxonomy, schema version을 고정했다.
  실제 provider 실패를 재현 가능한 로컬 evidence로 남기는 운영 계약이 생겼다.
- [[issues/BUI-177]]은 21:00 KST 기준 Reviewer 검토 중이다.
  Builder 완료 코멘트 기준 PR #99는 `test/plugin-packaging.test.js`를 추가해 plugin metadata, `plugins/built` symlink, skill-script 경로, provider 문서와 스크립트 포함 여부, README/package script 정합성을 57개 테스트로 고정했다.

## 드리프트 / 위험

- Queue Tick과 heartbeat가 프로젝트 진행을 유지하고 있지만, `active Queue Tick 부재` 같은 운영 복구 이슈가 반복되면 사람과 Coordinator의 수동 보정 부담이 커진다.
  이는 "사람은 의도와 제약 정의에 집중"한다는 목표와 부분적으로 충돌한다.
- BUI-260 검증은 BUI-279와 BUI-280을 파생시켰지만, 검증 자체의 KG 파일은 아직 없다.
  검증형 이슈가 PR/Finisher 없이 종료될 때도 durable evidence를 어떤 수준으로 남길지 기준이 더 필요하다.
- BUI-268과 BUI-174의 KG frontmatter에는 `merge_commit`이 비어 있다.
  PR 병합 후 Finisher가 KG merge commit을 보강하는 루프가 일부 파일에서 누락될 수 있다.
- BUI-177은 packaging smoke가 추가되었으나 Reviewer PASS와 Recorder KG 기록 전이다.
  plugin 배포 경로는 사용자-facing surface라 PR #99가 canonical PR로 유지되는지, 테스트 결과가 Reviewer에서 재확인되는지 추적해야 한다.
- BUI-280은 backlog에 남아 있다.
  `run.test.js` fake가 provider stdout fixture가 아니라 phase orchestration fake라는 구분이 명확하지 않으면, 향후 provider 계약 테스트 신뢰도 판단을 흐릴 수 있다.
