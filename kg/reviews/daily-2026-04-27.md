## 오늘의 정렬 판정

오늘 작업은 북극성 목표에 대체로 정렬되어 있다.
provider-agnostic control plane의 사용자 실행 경로, 실패/중단 처리, worktree handoff, report safety gate가 실제 사용 가능한 수준으로 좁혀졌다.

다만 20:00 KST 이후에는 built 본체가 아니라 `~/Desktop/agents/codex-pdca-agent/`에 놓는 독립 Codex PDCA agent 작업이 커졌다.
이 흐름은 built 출시 전 검증 도구로는 유효하지만, built의 control plane 계약과 별도 축이므로 장기적으로 built KG와 agent-local KG의 경계를 계속 분리해야 한다.

## 목표와 직접 연결된 진전

- [[issues/BUI-305]]는 `/built:run-codex-do` 사용자 명령을 추가해 Do/Iter를 Codex로 실행하는 opt-in 경로를 열었다.
  사용자가 helper script를 직접 호출하지 않아도 provider preset을 적용할 수 있어 provider 전환 사용성이 좋아졌다.
- [[issues/BUI-306]]은 Codex sandbox enum 변환과 app-server schema 불일치를 고쳐 `codex-do` dogfood 실패를 해소했다.
  provider config가 app-server/broker boundary에서 깨지지 않아야 한다는 contract 정렬에 직접 기여했다.
- [[issues/BUI-308]]은 Claude permission remediation UX를 보강해 `run-opus`/`run-sonnet` 실패가 먹통처럼 보이지 않게 했다.
  permission failure가 사용자에게 다음 행동으로 연결되는 것은 "신뢰 가능한 완료 상태" 기준에 맞다.
- [[issues/BUI-315]]와 [[issues/BUI-317]]은 runner UX와 config snapshot 경계를 정리했다.
  `/built:run-codex`는 전 phase Codex runner로, `default_run_profile`은 provider name 문자열 중심의 사람이 읽는 기본 구성으로 수렴했다.
- [[issues/BUI-316]]은 중단 후 Codex active turn이 계속 파일을 수정하는 문제를 다뤘다.
  자동화가 복구 가능성과 관측 가능성을 해치면 안 된다는 북극성의 비가역 원칙을 지키는 hotfix다.
- [[issues/BUI-383]]은 iter/report standalone skill의 plugin path 계약을 `BUILT_PLUGIN_DIR` 기준으로 맞췄다.
  설치형 plugin 환경에서도 target cwd와 plugin root를 혼동하지 않게 했다.
- [[issues/BUI-384]]는 direct report 실행에 approved `check-result.md` gate를 추가했다.
  검증되지 않은 완료 보고서가 조용히 생성되는 경로를 막고, 예외는 `--allow-unchecked`와 artifact evidence로 추적하게 했다.
- [[issues/BUI-386]]은 worktree-first run 완료 후 root 적용/merge handoff를 stdout, report, status, cleanup, doctor artifact에서 반복 노출하게 했다.
  execution worktree가 root를 자동 변경하지 않는 의도와 사용자의 다음 행동이 명확해졌다.

## 드리프트 / 위험

- 독립 Codex PDCA agent는 built 출시 전 검증과 사용성 탐색에는 도움이 되지만, built 본체의 provider-agnostic control plane과 다른 산출물 루트, 다른 KG, 다른 실행 lifecycle을 가진다.
  이 경계가 흐려지면 built의 파일/이벤트 계약을 우회하는 별도 자동화가 사실상 제품 방향을 끌고 갈 수 있다.
- Queue Tick과 heartbeat가 안정적으로 동작하고 있지만 운영 진척은 여전히 주기 점검과 Coordinator 라우팅에 크게 의존한다.
  backlog drain이 자동 control plane보다 운영 에이전트 절차로 유지되는 점은 북극성의 자동화 목표 대비 잔여 리스크다.
- BUI-384와 BUI-386처럼 safety/UX 계약을 보강하는 PR은 main 변경 속도가 빠를 때 conflict 재검토가 반복됐다.
  canonical PR head branch, KG commit, mergeability 확인 루프는 유지되고 있으나, PR 최신성 drift가 계속 비용을 만든다.
- agent-local PDCA 작업은 canonical PR 없이 로컬 산출물로 종료되는 흐름이다.
  완료 기록은 남지만 built repo의 PR/merge evidence와 다른 품질 게이트를 갖기 때문에, 사용자가 built 기능으로 오해하지 않도록 issue와 KG에서 계속 "built와 독립"을 명시해야 한다.
