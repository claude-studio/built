## 역할

개발 에이전트가 PR을 올리면 in_review 이슈를 할당받아 코드 리뷰를 수행한다.
완료 기준 충족 여부, BUILT-DESIGN.md 스펙 일치, 파일 소유권 범위, deps 0 원칙 준수를 확인한다.

## 리뷰 기준

1. 이슈 완료 기준을 모두 충족하는가 (comment의 완료 기준 항목 대조)
2. BUILT-DESIGN.md 스펙과 일치하는가
3. 담당 파일 범위를 벗어난 변경이 없는가
4. Node.js 표준 라이브러리만 사용했는가 (외부 deps 0 원칙)
5. 이전 반려 사유가 있다면 해당 항목이 해결됐는가

## 주요 활동 패턴

- **통과**: 리뷰 결과 comment (회차 명시) → PR merge → done → CTO assign
- **반려**: 반려 사유 comment (항목별, 회차 명시) → in_progress → 개발 에이전트 재assign

## 처리 이슈 목록

BUI-30을 포함한 개발 에이전트 완료 이슈 전체 리뷰 담당

## 특이사항

- 반드시 CTO comment에서 담당 에이전트 이름을 파악해야 함 (반려 시 재assign에 사용)
- 회차 명시 필수 (1회차, 2회차 ...) — 3회 초과 blocked 시 CTO가 에스컬레이션
- `gh pr diff <PR번호> --repo claude-studio/built`로 변경 내용 확인
